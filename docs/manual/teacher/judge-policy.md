# Judge 策略

> 本文档描述 VeriSpecOSLab 的自动评分 Judge 的配置策略。

## Judge 与 Pipeline 的关系

- **Pipeline**：高频运行（每次 push），提供开发反馈。测试结果公开。
- **Judge**：低频运行（阶段冻结后、最终提交后），计算正式评分。部分测试结果隐藏。
- **关系**：Pipeline 帮学生发现问题，Judge 量化最终成果。

## 测试分类

| 测试类型 | 可见性 | 运行频率 | 用途 |
|---------|--------|---------|------|
| 公开测试 (public) | 学生可见 | 每次 push | 开发反馈 |
| 生成测试 (generated) | 学生可见测试存在性，不可见具体内容 | 每次 verify | 验证实现是否符合 Spec |
| 隐藏测试 (hidden) | 学生不可见 | 仅 Judge 运行 | 评分的区分度 |
| 模糊测试 (fuzz) | 学生不可见 | 仅 Judge 运行 | 边界条件覆盖 |

## 评分公式

```text
Score = Σ(维度权重 × 维度得分)

其中：
  实现正确性得分 = 0.5 × public_pass_rate + 0.3 × hidden_pass_rate + 0.2 × generated_pass_rate
  验证覆盖度得分 = 0.4 × invariant_count_score + 0.3 × cross_component_score + 0.3 × goal_verification_score
```

## Judge 运行时机

1. **阶段冻结时**：每个阶段通过门禁后，Judge 对该阶段做一次评分快照。
2. **最终提交时**：Final Lab 提交后，Judge 做完整评分。
3. **申诉时**：学生申诉获批准后，Judge 重新运行。

## 隐藏测试策略

- 隐藏测试覆盖但不限于：
  - 资源泄漏场景（分配后不释放）
  - 并发竞争场景（多核下的锁正确性）
  - 边界值场景（最大值、最小值、零值）
  - 错误恢复场景（部分失败后的状态一致性）
- 隐藏测试不公开的原因：防止学生针对测试编写代码而非针对规格编写代码。

## 证据要求

每个评分项必须有关联证据（测试日志、不变量检查器输出、benchmark 数据）。Judge 不应基于无法复现的证据评分。

# 助教清单

> 本文档面向助教，提供每阶段的操作清单和常见问题处理流程。

## 学期开始前

- [ ] 熟悉 VeriSpecOSLab 的完整工作流（从建课到评分）
- [ ] 预演所有实验阶段，确认每个 Lab 卡片可执行
- [ ] 确认课程运行环境在目标平台上可正常运行
- [ ] 确认 QEMU + RISC-V 工具链版本与课程一致
- [ ] 阅读 [AI 审计策略](ai-audit-policy.md)，了解违规判定标准
- [ ] 建立 FAQ 初始内容
- [ ] 确认 Portal 的审核队列和通知系统正常

## 每阶段操作

### 阶段开始时

- [ ] 确认该阶段的实验卡片和 Book 章节已对学生可见
- [ ] 确认公开测试套件已部署
- [ ] 在课程论坛/频道发布阶段说明

### 阶段进行中

- [ ] 每日检查审核队列，处理设计审核请求
- [ ] 监控常见失败模式，更新 FAQ
- [ ] 回答学生问题（优先引用 Book 和 Specs 手册中的内容）

### 阶段结束时

- [ ] 确认所有学生已提交或标记为超时
- [ ] 处理补交请求
- [ ] 汇总本阶段常见问题，更新下一轮的改进建议

## 设计审核操作

### 审核 ArchitectureSeed

```text
检查项：
- [ ] 所有必填字段存在
- [ ] reference_systems 具体（非纯标签）
- [ ] goals 和 non_goals 并存
- [ ] constraints 具体可验证
审核结论：通过 / 需要修改（附具体修改建议）
```

### 审核 ArchitectureSlice

```text
检查项：
- [ ] mechanisms 描述具体
- [ ] depends_on 引用正确
- [ ] invariants 可验证
- [ ] validation_binding 对应实际测试
审核结论：通过 / 需要修改
```

## 异常处理

### 场景 1：自动验证失败但学生声称是基础设施问题

```text
处理流程：
1. 检查 Pipeline 日志，确认是否是 runner/infra 错误
2. 如果是 infra 错误 → 触发补跑，保留原始 run
3. 如果是学生错误 → 返回公开诊断，要求修正后重新提交
```

### 场景 2：学生请求阶段超时补交

```text
处理流程：
1. 确认超时原因（医疗/技术/其他）
2. 教师批准后，重置阶段状态为 active
3. 记录补交原因和时间
```

### 场景 3：疑似 AI 违规

```text
处理流程：
1. 对比代码 commit 和 AI 日志的时间线
2. 确认核心模块是否有对应的 ModuleSpec
3. 轻微可疑 → 提醒学生补充日志
4. 严重可疑 → 升级到教师，标记为需要答辩重点审查
```

## 评分操作

### 助教评分的维度

- 架构设计质量（15%）：审查 ArchitectureSeed/Slice/ADR
- AI 协作规范（10%）：审查 AI 日志
- 最终报告（10%）：审查最终报告

### 助教不评分的维度（平台自动评分）

- 规格完整性（20%）：平台检查
- 实现正确性（30%）：平台运行测试
- 验证覆盖度（15%）：平台运行不变量检查器

## 学期结束后

- [ ] 汇总学生常见错误和 FAQ，更新下学期的实验材料
- [ ] 反馈实验卡片和 Book 章节的改进建议
- [ ] 汇总 AI 审计结果，建议 AI 策略修订
- [ ] 归档评分数据和申诉记录

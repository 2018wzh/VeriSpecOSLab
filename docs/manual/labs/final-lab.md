## 1. 设计问题

你已经走完了 10 个阶段，从零构建了一个完整的操作系统。现在需要回答最后一个、也是最重要的问题：**你如何证明你的 OS 不只是"跑通了"，而是"正确地实现了你的设计意图"？**

Final Lab 不是"再做一个项目"。它是你全部工作的总结、审视和辩护。核心任务：

- **汇总证据**：将 10 个阶段分散的验证证据整合为可审查的完整证据链
- **追溯一致性**：从 ArchitectureSeed 出发，逐层验证设计意图是否贯彻到实现
- **记录诚实性**：展示设计演化过程，哪些变了、为什么变、怎么验证的
- **反思设计判断**：如果重来一次，你会做哪些不同的决策？

## 2. 设计空间

| 维度 | 你需要回答的问题 |
|------|----------------|
| 设计一致性 | 你的 ArchitectureSeed goals 是否在 10 个 ArchitectureSlice 中一致贯彻？non-goals 是否真的没有被实现？ |
| 实现正确性 | 你的验证覆盖了哪些层次？（功能测试 / 不变量检查 / 跨组件验证 / 设计一致性 / 演化追溯） |
| 演化诚实性 | 有多少次设计变更？每次变更是否通过 SpecPatch 记录？是否有 ADR 解释理据？ |
| 失败与反思 | 你遇到了哪些有意义的失败？通过什么手段发现的？学到了什么？ |

## 4. 提交物

### 4.1 最终报告（必做）

最终报告不是"把每个阶段的文档粘贴在一起"。它是一份独立的、面向不了解你过程的人的完整论述：

1. **你的 OS 是什么（1-2 页）** — ArchitectureSeed 核心目标、关键设计决策、独特剖面
2. **你的 OS 怎么运作的（3-5 页）** — 架构图 + 每个模块的职责和数据流
3. **你怎么验证它的（2-3 页）** — 不变量清单（摘要）、关键验证结果、跨组件验证
4. **它经历了什么演化（1-2 页）** — 关键设计变更（SpecPatch）+ 失败案例
5. **你学到了什么（≥500 字 Reflection）** — 具体的、可迁移的设计洞察

### 4.2 验证证据包（必做）

```
evidence/
├── verify-public.log       # vos verify public 完整输出
├── verify-full.log         # vos verify full 完整输出
├── invariants/             # 每个模块至少 1 条不变量检查器通过日志
├── cross-cutting/          # 跨组件不变量验证日志
└── stress/                 # 10 分钟稳定性运行日志
```

### 4.3 演化记录（必做）

- 至少 1 个 SpecPatch：记录设计变更的完整过程（原设计 → 变更理由 → 新设计 → 验证更新）
- 至少 1 个失败分析：现象 → 定位过程 → 根因 → 修复 → 修复后验证

### 4.4 答辩材料（必做）

- 一张架构图（模块 + 数据流 + 关键不变量标注）
- 关键设计决策的 ADR 摘要（1 页）
- 质量门禁结果一览表

## 5. 质量门禁

### 5.1 自动门禁
- [ ] `vos verify public` 全部通过
- [ ] `vos verify full` 全部通过
- [ ] 所有 spec 文件通过 `vos spec lint`

### 5.2 人工审查门禁

| 门禁 | 最低要求 |
|------|---------|
| 不变量覆盖 | ≥ 5 个不变量检查器 + ≥ 1 个跨组件不变量 |
| 失败分析 | ≥ 1 个完整的失败分析案例 |
| SpecPatch | ≥ 1 个有效的 SpecPatch |
| 最终报告 | 包含完整的 5 个部分 |
| Reflection | ≥ 500 字 |
| 架构图 | 清晰展示模块、数据流和不变量 |

### 5.3 自学者额外门禁
- [ ] 系统在 QEMU 中运行 10 分钟无崩溃、无内存泄漏
- [ ] 能从 seed.yaml 逐层追溯到实现代码和不变量检查器

## 6. 验收命令

```bash
bun run vos -- verify public
bun run vos -- verify full
bun run vos -- report generate --final
```

## 7. AI 使用边界

| 允许 | 不允许 |
|------|--------|
| AI 辅助审查最终报告的完整性和逻辑一致性 | AI 替你写 Reflection。反思必须是你的真实思考 |
| AI 辅助检查证据包的文件是否齐全 | AI 替你判断"验证是否充分" |
| AI 辅助生成架构图的 mermaid/dot 代码 | AI 替你决定哪些不变量画在图上 |

## 8. 自学者最终自检清单

如果你没有教师验收，以下是替代教师检查的自我评估清单。每一项都要诚实回答：

### 设计完整性
- [ ] ArchitectureSeed 中的每个 goal 都能在后续阶段找到对应的设计决策
- [ ] 每个 non-goal 在 10 个阶段中确实没有被意外实现
- [ ] 至少 3 个 ADR 记录了关键设计决策及其理由
- [ ] 至少 1 个 SpecPatch 诚实记录了设计变更

### 实现正确性
- [ ] 至少 5 个不变量检查器可运行且通过
- [ ] 至少 1 个跨组件不变量已在 CompositionSpec 中定义且可验证
- [ ] 系统运行 10 分钟不崩溃、不内存泄漏
- [ ] 至少 1 个 bug 是通过不变量检查器发现的（而非功能测试）

### 文档自洽性
- [ ] seed.yaml → ArchitectureSlice → ModuleSpec → 实现代码 → 不变量检查器，五层之间没有矛盾
- [ ] 能用一张图画出你的 OS 的完整架构（模块 + 数据流 + 不变量）
- [ ] 每个模块的 OperationContract 的 postcondition 能在不变量检查器中找到对应检查

### 失败记录
- [ ] 记录了至少 2 个"曾经失败、后来修复"的案例（现象 → 定位 → 根因 → 修复 → 验证）
- [ ] 至少 1 个案例通过不变量检查器发现（而非功能测试）

### 如果还有时间（加分项）
- [ ] 提供一个可启动的磁盘镜像（`qemu-system-riscv64 -drive file=myos.img`）
- [ ] 写一篇 500 字的 Reflection：如果重做，ArchitectureSeed 会有什么不同？
- [ ] 录制一个 3 分钟的演示视频

## 9. 常见错误与排查

| 症状 | 可能原因 | 排查方法 |
|------|---------|---------|
| 最终报告写成流水账 | 按时间顺序写"第 N 周做了什么"，而非按逻辑组织 | 重构：以设计问题为纲（"我的 OS 如何解决 X？"），而非以时间为纲 |
| 验证证据不完整 | 开发过程中没有持续收集不变量检查器通过日志 | 从现在开始重新跑一遍所有不变量检查器，收集输出 |
| Reflection 太空泛 | "我学到了很多关于操作系统的知识"，这不是反思 | 具体到某个设计决策："我选了方案 A，但后来发现它在场景 B 下有问题。如果重来，我会选方案 C。" |
| 架构图太模糊 | "一个大框写 'Kernel'，旁边写 'User Programs'" | 画出具体的模块（内存管理、进程管理、文件系统等），标出它们之间的数据流和共享状态 |
| 找不到 SpecPatch 案例 | 设计变更时直接改了代码和 spec，没走 SpecPatch 流程 | 现在回溯：找 git log 中 spec 文件的变化，补写 SpecPatch |
| `vos verify full` 不通过 | 隐藏测试未覆盖，或 AI 生成测试发现了边界情况 | 逐个失败用例排查，这本身就是有价值的失败分析素材 |

# 最终报告模板

最终报告是你在 VeriSpecOSLab 中的综合成果展示。它不是"操作记录"，而是"设计论证"——证明你的系统是正确的、你的设计决策是合理的、你的验证是充分的。

## 报告结构

### 1. 架构概述（约 2 页）

- 你的 OS 的名称和一句话描述
- 整体架构图（模块关系图）
- 技术栈（目标平台、编程语言、工具链）
- 参考系统：你借鉴了哪些系统？修改了什么？拒绝了什么？为什么？
- 关键设计哲学：你的 OS 最核心的 2-3 个设计原则

### 2. 阶段设计总结（每阶段约 1 页）

对每个完成的设计阶段：

- 阶段目标：你要解决什么设计问题？
- 关键决策：你做了什么选择？为什么？是否有 ADR 记录？
- 引入的机制：哪些模块/操作/不变量？
- 验证状态：质量门禁通过情况，关键测试结果摘要

### 3. 验证证据（约 2-3 页）

- 公开测试矩阵的通过情况
- 不变量检查器列表及其覆盖范围
- 跨组件不变量及其验证结果
- 个性化目标验证结果（如有）
- 关键验证日志的引用链接

### 4. 设计演化（约 1-2 页）

- SpecPatch 列表：哪些规格发生了演化？为什么？
- 选择一个最有代表性的设计变更，详细说明：
  - 变更前：原设计是什么样的？
  - 触发原因：什么现象或分析触发了这次变更？
  - 变更后：新设计是什么样的？
  - 影响范围：哪些模块/操作/不变量被影响？
  - 验证证据：变更后的回归测试结果

### 5. AI 协作总结（约 1 页）

- AI 使用的总体情况：主要用于哪些方面？
- 最有效的 AI 辅助案例（1-2 个）
- AI 出错或被拒绝的案例（1-2 个）
- AI 使用对设计/实现的影响评估

### 6. 失败分析（约 2-3 页）

选择 2-4 个曾失败的测试或验证场景：

- 现象：失败的表现是什么？
- 定位：如何定位到根因？用了什么调试手段？
- 原因：是设计错误还是实现错误？是哪个 Spec 缺失或错误？
- 修正：如何修正？是改 Spec 还是改代码？
- 证据：修正后的验证证据

### 7. 个性化目标（如有，约 1-2 页）

- 目标描述
- GoalValidationContract 摘要
- 验证结果：correctness_guard 是否通过？benchmark 是否达标？
- 设计与实现的独特之处

### 8. 反思与展望（约 1 页）

- 如果重新开始这个项目，你会做哪些不同的设计决策？
- 你的 OS 最大的优点和最大的局限是什么？
- 下一步你最想探索的方向是什么？

## 格式要求

- 使用 Markdown 格式
- 图表使用 Mermaid 或 ASCII art
- 代码片段使用 fenced code blocks 并标注语言
- 引用 Spec 文件时使用相对路径
- 引用验证日志时使用 `.vos/runs/` 中的相对路径

## 提交

- 文件名：`reports/final-report.md`
- 提交时确保所有引用文件存在且路径正确
- AI 协作日志作为独立文件提交：`reports/ai-collaboration-log.md`

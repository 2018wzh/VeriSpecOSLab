# 附录：OS 内核调试方法论

> 本附录不是 GDB 命令速查表（那个在 [GDB 使用指南](gdb-guide.md)）。这是一份**系统性的内核调试方法论**——从"系统崩溃了"到"找到根因"的完整思维框架。

## 调试金字塔

内核 bug 的排查成本和调试工具的威力成反比。越底层的工具越强大，但上手成本越高。你可以按以下金字塔从顶到底逐级深入：

```
         ┌──────────┐
         │  思维实验  │ ← 最快、最便宜。在脑子里模拟代码路径。
         ├──────────┤
         │  printf   │ ← 最常用。在关键路径加日志。
         ├──────────┤
         │  不变量    │ ← 最有价值。让系统在运行时持续自检。
         ├──────────┤
         │  二分法    │ ← 最系统。逐步缩小可疑代码范围。
         ├──────────┤
         │   GDB     │ ← 最精确。单步执行、检查寄存器和内存。
         ├──────────┤
         │   JTAG    │ ← 最后手段。CPU 级别的硬件调试。
         └──────────┘
```

## 一、思维实验：在你碰键盘之前

80% 的内核 bug 可以靠"在脑子里把代码走一遍"定位到大致范围。习惯在动手之前问自己：

1. **什么被改变了？** 是改了一个寄存器？一个全局变量？一个页表条目？
2. **改变发生时，系统在什么上下文中？** 进程上下文？中断上下文？哪个 CPU？
3. **改变是否被其他代码路径看到？** 有没有并发访问？有没有缓存一致性问题？

## 二、不变量驱动的调试

你从阶段 3 开始写的那些不变量检查器——它们不只是"作业要求"。它们是**自动化的 bug 探测器**。

### 原则：每个可疑的数据结构，先写一个检查器

与其在 GDB 中手动遍历一个链表查重复——不如写一个 10 行的检查器，在每次修改后自动运行。表面上看你花了 10 分钟写检查器——但省掉了 2 小时的 GDB 手工排查。

### 原则：检查器要在 bug 发生后的第一时间运行

如果你只在"系统看起来正常"的时候运行检查器，它的价值减半。最佳实践：在每次 `kalloc`/`kfree` 之后、在每次 `bread`/`brelse` 之后、在每次 `fork`/`exit` 之后运行轻量检查器。它们在 O(1) 或 O(n)（n 很小）的时间内完成，开销可接受。

### 示例：一个"反例驱动"的调试流程

1. 系统在运行 `ls` 后随机 panic
2. 你怀疑是 buffer cache 的 refcount 问题
3. 在 `brelse()` 的末尾添加检查：`assert(b->refcnt >= 0)`——refcount 不应该为负
4. 重新运行。检查器在 `b->refcnt == -3` 时触发
5. 你知道了：某个地方的 `brelse` 比 `bread` 多调用了。缩小了排查范围。

## 三、二分法定位：当你不确定哪段代码有问题

这是最系统、最不依赖直觉的定位方法。适用于"系统启动后 X 秒崩溃，但不知道是哪段代码导致的"场景。

### 方法

1. 在代码中找一个"中间点"——在堆分配器初始化之后、在进程调度器启动之前
2. 在这个点插入 `while(1){}`（硬停止）
3. 如果系统在硬停止之前崩溃 → bug 在前半段（启动→分配器）
4. 如果系统到达硬停止且不崩溃 → bug 在后半段（调度器→用户态）
5. 重复二分，直到你定位到具体的函数

### 变体：日志二分

如果你的崩溃需要很长时间才复现，用日志代替硬停止。在关键路径上插入递增的 marker（`[BOOT:1]`, `[BOOT:2]`, ...）。崩溃后，最后一个成功输出的 marker 告诉你崩溃发生在哪两个 marker 之间。

## 四、GDB 在 OS 调试中的进阶用法

### 跨越特权级的断点

GDB 默认不知道你的内核有 S-mode 和 U-mode 的区别。以下技巧在跨特权级调试时非常重要：

```gdb
# 在用户态的特定地址设断点（需要先加载用户程序的符号表）
(gdb) add-symbol-file user/hello 0x1000   # 假设 hello 的 .text 在 0x1000
(gdb) b *0x1010                            # 在 hello 的某条指令设断点

# 在 trap 入口设断点——捕获所有从用户态到内核态的切换
(gdb) b trap_entry
(gdb) commands
  > printf "Trap from U-mode: scause=%x, sepc=%x\n", $scause, $sepc
  > continue
  > end

# 条件断点：只在特定进程触发时停下
(gdb) b context_switch if $a0 == 5   # 只在切换到 PID=5 时停下
```

### 跨 ISA 的 GDB 差异

| 特性 | RISC-V | x86-64 | ARM64 |
|------|--------|--------|-------|
| 通用寄存器命名 | `x0`-`x31` (或 `zero`,`ra`,`sp`,...) | `rax`,`rbx`,... | `x0`-`x30`, `sp`,`lr` |
| 程序计数器 | `pc` | `rip` | `pc` |
| 栈指针 | `sp` | `rsp` | `sp` |
| CSR/系统寄存器 | `csrr` 读取，如 `$satp` | 内置寄存器 `$cr3` | 系统寄存器 `$ttbr0_el1` |
| 页表基址寄存器 | `satp` | `cr3` | `ttbr0_el1` / `ttbr1_el1` |

### 内核 panic 后的事后分析（Post-Mortem）

如果你的内核在 panic 时打印了 trapframe（保存的寄存器快照），GDB 可以用来重建崩溃现场的调用栈：

```gdb
# 从 trapframe 中恢复 sepc（崩溃时的 PC）
(gdb) info line *0x80001234     # sepc 对应的源码行
(gdb) disas 0x80001220,0x80001240  # 反汇编崩溃附近的指令

# 从 trapframe 中恢复 sp（崩溃时的栈指针）
(gdb) x/20gx 0x8000FF00         # 查看栈内容
(gdb) bt                         # 如果 GDB 能识别栈帧，可直接回溯
```

## 五、分类调试指南

### 内存破坏（Memory Corruption）

**症状**：随机崩溃、数据莫名其妙的变了、崩溃点离真正的 bug 很远。

**定位方法**：
1. **Poison pattern**：在 `kfree` 时将页填充为 `0xDEADBEEF`。如果之后在"正在使用"的数据结构中看到这个值 → use-after-free。
2. **Canary**：在关键数据结构的前后各放一个 magic number。定期检查——如果被改了 → 邻近的缓冲区溢出。
3. **缩小嫌疑范围**：关掉所有"非核心"代码（从最简单的内核配置开始），逐步加回功能，看谁引入崩溃。

### 竞态条件（Race Condition）

**症状**：bug 只在特定时序下触发，加一个 `printf` 可能就让它消失了（Heisenbug）。

**定位方法**：
1. **加延迟暴露竞态**：在可疑的代码路径之间插入 `for(volatile int i=0;i<100000;i++)`。如果 bug 复现率显著上升 → 你找对了地方。
2. **强制上下文切换**：在关键区域的中间手动调用 `yield()`，模拟中断在最不合适的时候发生。
3. **记录时序**：在共享数据的读写处打印 CPU 编号和 `mtime` 时间戳。事后分析时间线。

### 死锁（Deadlock）

**症状**：系统停止响应，但中断仍在触发（时钟 tick 在递增）。

**定位方法**：
1. **锁持有者检查**：在每个 spinlock 的 acquire 和 release 处记录"谁持有锁、在哪个 CPU 上"。死锁时打印所有锁的状态。
2. **锁顺序检查**：定义全局的锁获取顺序（如"先拿进程锁，再拿文件锁"）。在每次 acquire 时检查顺序是否被违反。
3. **看门狗**：一个定期重置的计时器。如果看门狗触发（说明主循环被阻塞），dump 所有 CPU 的栈回溯。

## 六、全 ISA 内核调试速查

### 所有 ISA 通用的"第一条调试指令"

当你面对一台完全陌生的目标机器时，按以下顺序建立调试能力：

1. **让一个 LED 闪烁（或用 GPIO toggle 替代 LED）**——证明代码在 CPU 上运行
2. **输出一个字节到 UART**——证明你对外设的控制是正确的
3. **输出一个字符串**——证明你的栈和 C 运行环境正常
4. **输出一个变量的值（`printf("x=%d\n", x)`）**——证明你的格式化输出正常
5. 只有这四步都通过后，才开始加 GDB stub 或 JTAG

### RISC-V 特有调试技巧

```gdb
# 在 S-mode trap handler 入口处打印 trap 信息
(gdb) b trap_from_kernel if $scause != 0x8000000000000005
# 0x8000000000000005 = timer interrupt (bit 63=1 表示中断, code=5)
# 排除时钟中断，只看"真正"的异常

# 检查 satp 寄存器的 MODE 域（Bare=0, Sv39=8, Sv48=9）
(gdb) p/x ($satp >> 60) & 0xF
```

### x86-64 特有调试技巧

```gdb
# 检查 CR3（页表基址）
(gdb) p/x $cr3

# 解码 page fault 的 CR2（错误地址）
# CR2 在 page fault 发生时保存触发 fault 的虚拟地址
(gdb) p/x $cr2

# 检查中断标志（IF）
(gdb) p/x $eflags & 0x200   # Bit 9 = IF
```

### ARM64 特有调试技巧

```gdb
# 检查异常级别（EL）
(gdb) p/x ($cpsr >> 2) & 0x3   # 0=EL0, 1=EL1, 2=EL2, 3=EL3

# 检查页表基址
(gdb) p/x $ttbr0_el1           # 用户页表
(gdb) p/x $ttbr1_el1           # 内核页表

# 检查异常综合征寄存器（ESR_EL1：发生了什么异常）
(gdb) p/x $esr_el1
# EC 域（bits 31:26）：
#   0b010101 = SVC (syscall)
#   0b100100 = Data Abort (page fault)
#   0b100101 = Data Abort (permission fault)
```

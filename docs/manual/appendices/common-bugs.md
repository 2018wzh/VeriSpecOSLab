# 常见错误与调试

本文档汇总了 OS 内核开发中的常见错误类型和调试思路。错误归为设计错误和实现错误两类——前者是你的设计本身有问题（Spec 需要修正），后者是你的实现未正确反映设计（代码需要修正）。

## 分类框架

| 错误类型 | 症状 | 定位方法 | 修正方式 |
|---------|------|---------|---------|
| 设计错误 | 实现正确但系统行为不符合预期 | 回到 Spec 层检查前置条件、不变量 | 修改 ArchitectureSlice / ModuleSpec / ADR |
| 实现错误 | 实现与 Spec 描述的行为不一致 | 对比 Spec 中的前后置条件与代码行为 | 修改代码 |
| 缺失规格 | 实现了一个行为但 Spec 中没有描述 | 验证时发现"未指定的行为" | 补充 ModuleSpec / OperationContract |
| 规格冲突 | 两个 Spec 描述的行为互相矛盾 | 两个测试都声称通过但不可能同时为真 | 修改 CompositionSpec 解决冲突 |

## 按阶段分类的常见错误

### 启动阶段

- **症状**：QEMU 启动后无任何输出
- **可能原因**：链接脚本入口地址错误、UART 未初始化、栈未正确设置、BSS 未清零导致 C 代码异常
- **定位**：使用 GDB 在 `_start` 设断点，单步确认每条指令执行正确

- **症状**：输出乱码
- **可能原因**：UART 波特率配置错误、串口参数不匹配

### 内存管理阶段

- **症状**：分配器返回相同页
- **可能原因**：freelist 指针损坏、并发分配未加锁
- **定位**：在 `kalloc` 和 `kfree` 中加日志，追踪每页的分配/释放历史

- **症状**：启用分页后立即崩溃
- **可能原因**：页表项权限位设置错误、未建立 identity mapping、satp 写入方式不正确
- **定位**：在启用分页的指令前后设断点，检查 satp 值和页表内容

- **症状**：不变量检查器报错
- **可能原因**：你的不变量定义正确但实现有问题，或你的不变量定义本身有漏洞。先确认不变量是否合理，再查实现。

### 中断与 Trap 阶段

- **症状**：中断发生后系统崩溃
- **可能原因**：stvec 指向无效地址、trap handler 未保存/恢复全部寄存器、中断嵌套导致栈溢出
- **定位**：在 trap handler 入口设断点，检查寄存器状态

- **症状**：时钟中断不触发
- **可能原因**：sie 未使能 timer 中断、mtimecmp 设置不正确、PLIC 未正确配置

### 进程与调度阶段

- **症状**：fork 后子进程崩溃
- **可能原因**：页表复制不完整、trapframe 未正确初始化、栈设置错误
- **定位**：在子进程的第一条指令设断点

- **症状**：调度器不切换进程
- **可能原因**：时钟中断未触发 yield、scheduler 循环条件错误

### 文件系统阶段

- **症状**：写入后读出数据不一致
- **可能原因**：buffer cache 未正确回写、日志未正确提交
- **定位**：在 bread/bwrite 加日志，追踪每个块的读写

- **症状**：崩溃后文件系统损坏
- **可能原因**：日志恢复逻辑错误、超级块未正确更新

## 通用调试技巧

1. **先看 Spec，再看代码。** 遇到 bug 时，先确认是设计问题还是实现问题。
2. **缩小范围。** 用二分法注释掉部分代码，定位出错的精确位置。
3. **加不变量检查。** 在关键路径上插入 assert，让系统在违反不变量时立即停止。
4. **记录日志。** `printk` 是最原始但最可靠的调试手段。在中断上下文中小心使用。
5. **使用 QEMU 的日志功能。** `-d cpu_reset,int,mmu` 可以输出 CPU 状态变化和 MMU 操作。

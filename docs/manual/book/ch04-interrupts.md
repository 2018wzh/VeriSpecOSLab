# 第 4 章：中断与设备驱动 — 响应外部世界

## 4.1 本章要解决的问题

到目前为止，你的内核只能按预定的顺序执行代码——它无法"被打断"。中断机制改变了这一点：硬件可以在任何时候通知 CPU"有事情发生了"，CPU 暂停当前任务，进行处理，然后回到原来的任务。

## 4.2 中断 vs 轮询——一段争夺注意力的历史

在中断被发明之前，CPU 通过**轮询**（polling）检查设备状态。内核每隔一段时间逐个询问每个设备："你有数据吗？""你准备好了吗？"如果有，处理；如果没有，问下一个。

轮询的优点是简单——代码是一条直线。缺点是浪费——CPU 把大量时间花在了"问一个不忙的设备"上。更严重的是延迟问题——如果设备在两次轮询间隔之间产生了数据，那个数据要等到下一次轮询周期才会被处理。

**中断的诞生**把控制关系反转了。设备主动发出信号，打断 CPU 的当前工作。CPU 暂停手头的事，保存现场，处理设备请求，然后恢复。CPU 利用率大幅提升，响应延迟大幅降低。

但中断也带来了一种全新的 bug 类别：中断处理程序不在任何进程的"上下文"中运行。它不能睡眠、不能调用任何可能阻塞的函数。它和"普通的内核代码"之间共享的数据结构需要特殊的保护。

### 不同 ISA 的中断控制器对比

你的中断子系统设计受 ISA 选择的影响。以下是对比：

| 维度            | RISC-V PLIC                          | ARM GICv3                               | x86 APIC/x2APIC               |
| --------------- | ------------------------------------ | --------------------------------------- | ----------------------------- |
| 复杂度          | 低——平台级，所有中断源统一管理     | 中——CPU interface 和 distributor 分离 | 高——per-CPU LAPIC + IO-APIC |
| 中断 claim      | `plic_claim()` 读寄存器返回 IRQ 号 | `ICC_IAR0_EL1` 读寄存器返回 INTID     | LAPIC 的 ISR 寄存器           |
| 中断 complete   | `plic_complete()` 写寄存器         | `ICC_EOIR0_EL1` 写寄存器              | LAPIC 的 EOI 寄存器           |
| IPI（核间中断） | 通过 SBI 或 CLINT                    | GIC 的 SGI                              | LAPIC 的 ICR 寄存器           |
| 优先级          | 可配置优先级                         | 支持优先级+抢占                         | 任务优先级+向量优先级         |

RISC-V 的 PLIC 是最简单的起点。ARM 的 GIC 在工业中更常见。x86 的 APIC 历史包袱最重，不推荐作为第一个中断控制器的学习目标。不管你用哪个，中断处理的设计原理——保存上下文、快速处理、bottom-half 延迟——是跨 ISA 通用的。

## 4.3 中断上下文的铁律

中断处理程序运行在一种特殊的上下文中。三条铁律：

**不能睡眠。** 中断处理程序不在任何进程的上下文中——没有"当前进程"可以调度出去。如果你在中中断里调用了 `sleep()` 或获取了 sleeplock，行为是未定义的（通常会 deadlock 或 kernel panic）。

**必须快。** 中断处理程序运行期间，同级或更低优先级的中断通常被禁用。如果你的 handler 花了 10ms，其他设备的中断就延迟了至少 10ms。实际做法是把紧急部分放 handler（标记、读寄存器、调度一个 bottom-half），把耗时部分推迟到进程上下文。

**锁必须用 spinlock，且要关中断。** 如果一个数据结构同时被进程上下文和中断上下文访问，进程上下文在获取 spinlock 之前必须先禁用中断——否则可能出现"进程持有锁时中断进来，中断 handler 等同一个锁"的死锁。

## 4.4 设计维度

### 维度 1：中断路由

从硬件中断信号到内核中断处理函数，中间经过多层路由：

- **中断控制器**（如 RISC-V PLIC）：管理多个中断源的优先级和分发
- **CPU 中断使能**：`sie`/`sstatus.SIE` 控制 CPU 是否响应中断
- **中断委托**：哪些中断由 M-mode 处理？哪些委托给 S-mode？

你需要回答的问题：

- 你的中断控制器如何初始化？每个中断源的优先级如何设置？
- 你的系统中有哪些中断源？（时钟、UART、可能的磁盘、可能的外部中断）
- 中断向量如何定位？`stvec` 指向直接处理函数还是分发表？

### 维度 2：时钟中断

时钟中断是操作系统调度的心脏。你需要决定：

- **时钟源**：RISC-V 的 `mtime`/`mtimecmp` 是标准时钟源
- **Tick 粒度**：每个 tick 间隔多少时间？10ms？1ms？还是动态 tick？
- **Tick 处理**：每个 tick 做什么？至少递增系统时间计数器。在引入进程调度后，还涉及时间片检查。

你需要回答的问题：

- Tick 粒度的选择如何影响系统的响应性和开销？
- 下一个 tick 如何设置？在 tick handler 中设置 `mtimecmp` 为 `mtime + interval` 是最常见的做法。

### 维度 3：设备发现

你怎么知道系统中有哪些设备、它们在什么地址？

- **设备树（DTB）**：QEMU 生成设备树描述所有外设。你可以解析 DTB 动态获取设备信息。
- **ACPI**：x86 平台上的设备描述标准。RISC-V 教学场景中通常不使用。
- **硬编码**：直接硬编码已知设备的 MMIO 地址。简单但移植性差。

你需要回答的问题：

- 设备发现方式的移植性对你的 OS 有多重要？如果只支持 QEMU `virt`，硬编码是可行的。如果计划移植到真实硬件，DTB 解析是必要的。
- 如果你用硬编码，至少要在 ADR 中说明原因，并在代码中集中管理这些地址（便于将来修改）。

### 维度 4：设备驱动模型

你需要决定设备驱动的组织方式：

- **驱动注册**：设备驱动如何向内核注册自己？通过什么接口？
- **中断处理程序注册**：每个 IRQ 如何绑定到对应的处理函数？
- **驱动与内核的接口**：驱动提供什么接口给内核的其他部分？（如 UART 驱动提供 `uart_read`/`uart_write`）

你需要回答的问题：

- 你的驱动模型如何处理设备热插拔？在教学 OS 中通常不需要，但你应该知道这个限制。
- 一个设备对应一个驱动，还是一类设备共享一个驱动？

### 维度 5：中断上下文 vs 进程上下文

这是内核编程中最容易出错的维度：

- **中断上下文**：中断处理程序运行时，不在任何进程的上下文中。不能睡眠、不能调用可能阻塞的函数。
- **进程上下文**：内核代码在某个进程的上下文中执行（如 syscall 处理函数）。可以睡眠、可以获取 sleeplock。
- **锁的规则**：如果要同时从中断上下文和进程上下文访问某个数据结构，必须用 spinlock（中断安全的），不能用 sleeplock。

你需要回答的问题：

- 哪些数据结构会被中断上下文访问？它们的锁策略是什么？
- 中断嵌套是否允许？（即处理一个中断时是否允许被另一个中断打断）

### 维度 6：DMA 分配

DMA（直接内存访问）允许设备直接读写物理内存，绕过 CPU。DMA 需要物理连续的内存缓冲区。

在阶段 4，如果你不涉及 virtio 磁盘或复杂 DMA 设备，DMA 分配可以暂时不做。但你需要知道它的存在和基本要求。

### 维度 7：多核下的中断处理

阶段 2 你已经处理了多核启动。在阶段 4，多核对中断子系统提出了新要求：

- **per-HART trap 入口**：每个 HART 需要独立的 `stvec` 设置——指向各自的 trap handler 或共享 trampoline 的不同入口。每个 HART 在内核态需要独立的内核栈来处理中断和异常。
- **IPI（Inter-Processor Interrupt）**：CPU 间中断是多核协调的基础机制。通过 SBI 发送 IPI，一个 CPU 可以通知其他 CPU 执行 TLB 刷新、重新调度、或处理特定的跨核请求。RISC-V 通过 CLINT 或 SBI `sbi_send_ipi()` 实现。
- **中断亲和性**：哪些中断由哪个 HART 处理？在简单实现中，可以让所有外部中断路由到 HART 0。更复杂的实现需要配置 PLIC 的中断上下文（hart context），允许不同中断绑定到不同 HART。
- **中断禁用粒度**：单核下 `intr_off()` 全局禁用中断即可。多核下，`intr_off()` 只禁用本 CPU 的中断——其他 CPU 仍然可以接收中断并进入 trap handler。共享数据结构的保护需要 spinlock + `intr_off()` 的组合。

你需要回答的问题：

- PLIC 的 hart context 如何配置？每个 HART 需要独立的 claim/complete 寄存器区域吗？
- 哪些中断是 per-HART 的（如时钟中断通常每个 HART 独立接收），哪些是全局的（如 UART 中断通常只路由到一个 HART）？

## 4.5 典型设计路线（参考）

### 中断处理流程

```text
硬件中断 → PLIC → CPU 检查 sie/sstatus → 跳转 stvec → trap_entry(asm)
    → 保存寄存器 → C 中断分发函数 → 识别中断源（读 scause / PLIC claim）
    → 调用对应的 IRQ handler → PLIC complete → 恢复寄存器 → sret
```

### 驱动注册模型

一种通用模式是中断处理程序注册表：

```c
// 概念性示例，非具体实现
void register_irq_handler(int irq, void (*handler)(void));
void dispatch_irq(int irq);
```

每个设备驱动在初始化时注册自己的中断处理函数。

## 4.7 ⚡ 挑战：中断嵌套、Bottom-Half 与通用 IRQ 框架

### 挑战 A：中断嵌套

在基础实现中，处理一个中断时同优先级的中断被 CPU 自动禁用。但高优先级中断是否可以抢占低优先级中断的处理？

**中断嵌套的关键要求**：

1. 进入中断时保存 `sstatus.SIE` 的旧值
2. 在处理低优先级中断的过程中，手动重新开启中断使能（仅在确认安全后）
3. 嵌套中断的栈使用必须可预测——每个嵌套级别增加固定大小的栈帧
4. 嵌套深度有硬上限（防止栈溢出）

**验证**：测量最大中断禁用区间。在嵌套模型中，这个区间应该是"单个 handler 中显式关中断的区间"而不是"整个 handler 的执行时间"。

### 挑战 B：Bottom-Half 机制

中断 handler 必须快，但有些处理天然需要时间。Linux 的解决方案是将中断处理分为两半：

- **Top-half（硬中断上下文）**：读设备寄存器、应答中断、调度 bottom-half。必须在 < 10μs 内完成
- **Bottom-half（进程上下文或软件中断）**：实际数据处理、网络包解析、磁盘 I/O 完成处理。可以睡眠

在你的教学 OS 中，最简单的 bottom-half 是"设置一个标志位，让调度器在下次调度时检查"：

```text
IRQ handler:
  读设备寄存器
  设置 bottom_half_pending = true
  PLIC complete
  返回

调度器（每次 schedule() 调用时）:
  if (bottom_half_pending) {
    执行延后的处理
    bottom_half_pending = false
  }
```

**验证**：用 `mtime` 精确测量 top-half 的执行时间（从 PLIC claim 到 PLIC complete），目标 < 10μs。

### 挑战 C：通用 IRQ 注册框架

不要在代码中硬编码"IRQ 10 = UART"。实现一个通用的 IRQ 注册表：

```text
irq_register(irq_num, handler_func, handler_name, dev_id)
irq_unregister(irq_num, dev_id)
irq_dispatch(irq_num)  // 查找并调用注册的 handler
```

**这个框架的长期价值**：当你在阶段 8 添加 USB 或 PCI 驱动时，你只需要 `irq_register()` 新设备的中断 handler——不需要修改中断子系统的核心代码。这是你的 HAL 和驱动模型之间的关键接口。

**验证**：注册两个不同的 handler 到不同的 IRQ，通过 QEMU monitor 注入中断，验证正确的 handler 被调用。

## 4.8 质量门禁

### 中断控制器门禁

- [ ] PLIC 初始化后，至少一个中断源可被正确路由和处理
- [ ] 中断处理后 PLIC complete 被调用，中断源被清空

### 时钟中断门禁

- [ ] 时钟中断周期性触发
- [ ] 系统时间计数器在增长
- [ ] Tick 间隔与设计一致

### 设备中断门禁

- [ ] 至少一个设备（UART）通过中断驱动方式工作
- [ ] 接收中断可正确触发数据读取
- [ ] 发送中断可正确驱动数据发送

### 稳定性门禁

- [ ] 在大量中断负载下（如持续接收串口数据），系统不崩溃
- [ ] 中断处理不破坏进程上下文的内核数据结构

## 4.9 常见陷阱

1. **中断重入**：中断处理程序执行期间如果又来了同类型中断。大多数平台自动禁用同优先级中断，但确认你的平台行为。
2. **忘记 PLIC complete**：不调用 complete 会导致中断控制器不发送后续中断。
3. **中断处理程序过大**：中断处理程序应尽可能短。将复杂处理推迟到进程上下文（如 Linux 的 bottom half 机制）。
4. **在中断上下文中使用 sleeplock**：这会导致死锁或未定义行为。
5. **设备树解析错误**：DTB 格式有严格的规范。建议使用成熟的 DTB 解析库或参考代码，不要从零实现。

## 4.10 与前后阶段的接口

- **依赖阶段 3**：MMIO 区域需要通过页表映射；DMA（如适用）依赖物理页分配器
- **为阶段 5 提供**：时钟中断将驱动进程调度器的 tick；中断框架将在阶段 5 扩展为用户态 trap 处理

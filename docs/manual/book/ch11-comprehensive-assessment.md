# 第 11 章：综合验收 — 从部件到整体

> **对应实验**：[Final Lab: 综合验收与答辩](../labs/final-lab.md)

## 11.1 本章要解决的问题

你已经走完了 10 个阶段。你的 OS 能从硬件启动到用户 Shell、能管理内存和文件、能响应中断和调度进程、有独特的剖面和验证体系。现在面临最后一个问题：**如何证明这一切不只是"跑通了"，而是"有设计意图地运作着"？**

Final Lab 不是"再做一个项目"。它是你全部工作的总结、审视和辩护。本章帮你理解总结什么、怎么总结、用什么标准检验自己的总结。

## 11.2 综合验收的四个维度

### 维度一：设计一致性

你的 ArchitectureSeed 中的 goals 是否在后续所有 ArchitectureSlice 中得到了一致的贯彻？你的 non-goals 是否确实没有在实现中悄悄出现？

检查方法：从 seed.yaml 出发，逐条 goal 追溯：它在哪个 ModuleSpec 中体现？对应的不变量是什么？实现代码中哪段逻辑支撑了它？

### 维度二：实现正确性

你的 OS 通过了哪些测试？你的不变量检查器覆盖了哪些危险场景？有没有跨组件的不变量在运行中被持续验证？

检查方法：逐模块列出不变量 → 逐条不变量找到对应的检查器源码 → 逐条确认有通过的运行日志。

### 维度三：演化诚实性

你的设计在 10 个阶段中改变了多少次？每次改变是否有 SpecPatch 记录？是否有 ADR 记录了改变的理据？

检查方法：`spec/evolution/` 下的 SpecPatch 数量和内容 → 确认每个变更都有理由、有影响分析、有验证更新。

### 维度四：失败记录与反思

你遇到了哪些失败？你是怎么发现根因的？你学到了什么？如果重来一次，你会做什么不同的选择？

检查方法：`spec/evolution/` 下的 failure analysis 案例 → Reflection 的质量。

## 11.3 Final Lab 提交物详解

### 11.3.1 最终报告

最终报告不是"把每个阶段的设计文档粘贴在一起"。它是一份独立的、完整的论述，应包含：

1. **你的 OS 是什么（1-2 页）** — ArchitectureSeed 中的核心目标、关键设计决策、独特剖面
2. **你的 OS 怎么运作的（3-5 页）** — 一张架构图 + 每个模块的职责和数据流
3. **你怎么验证它的（2-3 页）** — 不变量清单（摘要）、关键验证结果、跨组件验证
4. **它经历了什么演化（1-2 页）** — 关键的设计变更（SpecPatch）和失败案例
5. **你学到了什么（500 字 Reflection）** — 不只是"我学到了很多"，而是具体的、可迁移的设计洞察

### 11.3.2 验证证据包

- `vos verify public` 完整输出
- `vos verify full` 完整输出
- 每个模块至少 1 条不变量检查器通过日志
- 至少 1 条跨组件不变量的验证日志

### 11.3.3 答辩材料

- 架构图（模块 + 数据流 + 不变量标注）
- 关键设计决策的 ADR 摘要
- 质量门禁结果一览表

## 11.4 自学者最终自检清单（扩展版）

### 设计完整性
- [ ] ArchitectureSeed 中的每个 goal 都能在后续阶段找到对应的设计决策
- [ ] 每个 non-goal 在 10 个阶段中确实没有被意外实现
- [ ] 至少 3 个 ADR 记录了关键设计决策及其理由
- [ ] 至少 1 个 SpecPatch 诚实记录了设计变更

### 实现正确性
- [ ] 至少 5 个不变量检查器可运行且通过
- [ ] 至少 1 个跨组件不变量已在 CompositionSpec 中定义且可验证
- [ ] 系统运行 10 分钟不崩溃、不内存泄漏
- [ ] `vos verify public` 全部通过

### 文档自洽性
- [ ] seed.yaml → ArchitectureSlice → ModuleSpec → 实现代码 → 不变量检查器，五层之间没有矛盾
- [ ] 能用一张图画出你的 OS 的完整架构（模块 + 数据流 + 不变量）

### 失败记录
- [ ] 记录了至少 2 个"曾经失败、后来修复"的案例（现象 → 定位 → 根因 → 修复 → 验证）
- [ ] 至少 1 个案例是通过不变量检查器发现的（而非功能测试）

### 如果还有时间
- [ ] 提供一个可启动的磁盘镜像（`qemu-system-riscv64 -drive file=myos.img`）
- [ ] 写一篇 500 字的 Reflection：如果重做，ArchitectureSeed 会有什么不同？
- [ ] 录制一个 3 分钟的演示视频

## 11.5 质量门禁

| 门禁 | 要求 | 验证方式 |
|------|------|---------|
| 公开验证 | `vos verify public` 全部通过 | 自动 |
| 完整验证 | `vos verify full` 全部通过 | 自动 |
| 不变量覆盖 | ≥ 5 个不变量检查器 + ≥ 1 个跨组件不变量 | 人工审查 |
| 失败分析 | ≥ 1 个完整的失败分析案例 | 人工审查 |
| SpecPatch | ≥ 1 个有效的 SpecPatch | 人工审查 |
| 最终报告 | 完整的 5 部分报告 | 人工审查 |
| Reflection | 500 字以上的反思 | 人工审查 |

## 11.6 常见陷阱

**把最终报告写成日志。**"第 1 周我做了 X，第 2 周我做了 Y..." 这不是报告，是流水账。报告面向一个不了解你过程的人，解释你的设计是什么、为什么这样做、怎么验证的。

**验证证据不足。**"系统能跑 Shell"是一个 demo，不是验证。你需要的是：不变量检查器的通过日志、测试套件的通过输出、跨组件不变量的验证结果。

**失败分析太浅。**"内存分配器崩溃了，我改了 freelist 的指针，好了。" 这不是分析。分析要回答：崩溃时的具体状态是什么？违反了哪个不变量？根因是什么？为什么之前的检查器没发现？

**Reflection 太空。**"通过这门课我学会了操作系统。" 这不是反思。好的反思是具体的："我在阶段 3 选了 buddy system，但阶段 5 发现它让连续物理页分配变得困难。如果重来，我会在阶段 3 同时考虑阶段 5 的需求。"

**没有架构图。**"我的 OS 太简单了不需要画图"。即使是最小的教学 OS 也有模块交互。一张好的架构图能让你自己看清楚模块间的依赖关系，也能让审查者快速理解你的设计。

## 11.7 验收命令

```bash
bun run vos -- verify public
bun run vos -- verify full
bun run vos -- report generate --final
```

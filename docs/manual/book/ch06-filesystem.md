# 第 6 章：文件系统 — 持久化

## 6.1 文件系统简史——从 FAT 到 ZFS

文件系统的发展史，本质上是对两个问题的持续回答：**数据怎么组织在磁盘上？系统崩溃后怎么恢复？**

但在回答这两个问题之前，有一个更根本的问题需要先被回答：**"文件"这个抽象是什么时候出现的？**

### MULTICS（1965）——第一个层级文件系统

今天你认为是"理所当然"的东西——文件有名字、文件在目录里、目录可以嵌套——其实都是 MULTICS（"Multiplexed Information and Computing Service"）在 1965 年第一次系统化实现的。在此之前，大多数系统把"文件"看作磁带或磁盘上的一段连续数据块，通过数字编号引用——没有名字，没有目录，没有层次。

MULTICS 引入了三个至今通用的概念：① **层级目录树**——文件被组织在一棵倒置的树中，根目录 `/` 是起点，每个目录可以包含文件或子目录；② **符号名称**——用户通过路径名（如 `/usr/ken/notes.txt`）访问文件，而不是通过物理地址；③ **访问控制列表（ACL）**——每个文件有自己的权限列表，定义哪些用户可以读、写、执行。

MULTICS 的层级文件系统在当时是革命性的——它把"数据怎么组织"从物理问题（磁盘扇区）变成了逻辑问题（路径名）。Ken Thompson 和 Dennis Ritchie 在离开 MULTICS 项目后把这三个概念几乎原封不动地带到了 Unix 中。你今天在终端里敲 `mkdir` 和 `ls` 的时候，用的是 MULTICS 在 1965 年定义的范式。

> **原始文献：** R. C. Daley and P. G. Neumann, "A General-Purpose File System For Secondary Storage," *Proceedings of the AFIPS Fall Joint Computer Conference*, pp. 213-229, November 1965. 这篇论文描述了 MULTICS 文件系统的完整结构——包括层级目录、符号链接和 ACL——所有的概念都在这里了。

### Unix 文件系统（1974）——MULTICS 的精简版

Unix 把 MULTICS 层级文件系统做到了极致简洁。和 MULTICS 的繁复 ACL 不同，Unix 只用三个 bit——owner/group/other 各一个 `rwx`——总共 9 位。不够安全，但够简单。Unix 去掉了 MULTICS 的"段"概念，把一切统一在字节流模型下——一个文件就是一串字节，内核不关心字节的含义。这让 `read()` 和 `write()` 适用于一切。

### FAT（1977）——最简单的东西能走多远

1977 年，微软为 8 位机设计了 FAT。它的故事和 Unix 完全不同——不是从大型机的学术研究中演化而来的，而是 Bill Gates 和 Marc McDonald 在一周之内为 8086 处理器手写的一个"勉强能用"的磁盘格式。

FAT 的核心结构极其朴素：磁盘开头是一张"文件分配表"（FAT），每个文件对应 FAT 中的一条簇链。目录项存文件名和起始簇号。读取文件就是沿着 FAT 链逐个读簇。FAT 的简单性使得它几乎被所有操作系统支持——到今天，FAT32 依然是 U 盘和 SD 卡的事实标准。但它不支持权限、没有 journal、大文件效率低。

FAT 的历史教训是：**一个简单的设计，即使技术上不优雅，只要它能被所有系统互操作地读写，就会存活几十年。** FAT 不是最好的文件系统——但它是最兼容的。

### FFS（1984）——当软件开始理解硬件

1984 年，Berkeley 的 Marshall Kirk McKusick 在重新设计 Unix 文件系统时，意识到了一个被之前所有文件系统忽略的问题：**磁盘不是随机访问存储器——它有物理结构。** 磁头从磁盘的一个位置移动到另一个位置需要时间（寻道延迟），磁盘旋转到目标扇区也需要时间（旋转延迟）。如果文件数据散布在磁盘各处，读取一个文件需要磁头在磁盘上来回奔波——性能灾难。

McKusick 的 Fast File System（FFS）引入了两个关键思想。第一，**柱面组（cylinder group）**——把相关的数据和 inode 放在磁盘上物理相邻的磁道上，减少磁头移动。第二，**块大小从 512 字节增加到 4096 字节**——减少读取同一个文件需要的 I/O 操作次数。FFS 让同一个磁盘和同一个硬件上的文件系统吞吐量提高了 10 倍——没有改变任何硬件，只是改变了数据在磁盘上的摆放方式。

> **原始文献：** M. K. McKusick, W. N. Joy, S. J. Leffler, and R. S. Fabry, "A Fast File System for UNIX," *ACM Transactions on Computer Systems*, vol. 2, no. 3, pp. 181-197, August 1984. 这篇 17 页的论文改变了文件系统设计的方向——从"数据结构驱动的设计"变成了"硬件感知的设计"。你的教学 OS 中的 inode、间接块、superblock 概念，在这篇论文中都有明确的定义。

**ext 家族——从"够用"到"工业级"。** ext2（1993）是 Linux 的第一个实际使用的文件系统。它是 FFS 的简化版——去掉了柱面组的复杂布局，保留了 inode、目录和间接块。ext3（2001）加了一层写前日志（journal），解决了"崩溃后 fsck 要跑几十分钟"的问题。ext4（2008）扩展到支持 1 EB 的卷和 16 TB 的文件。

**ZFS 和 Btrfs——文件系统的"终极形态"。** Sun 的 ZFS（2005）和 Linux 的 Btrfs 引入了两个革命性的概念。第一个是**写时复制（Copy-on-Write）**——从未原地覆写数据，永远是写到新位置，然后原子地更新指针。这自然支持快照、回滚和持续数据校验。第二个是**数据校验和（checksum）**——每个数据块存储自己的校验和，读的时候验证，发现静默损坏就自动修复（如果有冗余副本）。ZFS 的"永不信任磁盘"哲学代表了对存储可靠性最悲观的假设。

### 对你这门课的意义

你的教学 OS 不需要 ZFS 级别的功能。但你在选择文件系统布局时——inode vs FAT、日志 vs 无日志、直接块 vs 间接块——是在和上述历史对话。了解 FAT 的极简主义和 FFS 的磁盘感知布局，能帮你在简单性和功能性之间做出有意识的取舍。

## 6.2 设计维度

### 维度 1：存储设备与驱动

你的文件系统需要一个存储设备：

- **Virtio 磁盘**：QEMU `virt` 机器提供 virtio-blk 设备。需要通过 MMIO 配置 virtio 队列，使用 DMA 传输数据。
- **其他设备**：NVMe、AHCI 等。教学 OS 建议用 virtio-blk，因为它在 QEMU 中有良好支持且接口相对简单。

你需要回答的问题：
- 你的磁盘驱动如何与文件系统层交互？提供什么接口？（`disk_read(block_no, buf)` / `disk_write(block_no, buf)`）
- 磁盘的扇区大小和块大小各是多少？你的文件系统使用的逻辑块大小是多少？

### 维度 2：Buffer Cache

直接读写磁盘太慢。Buffer Cache 在内存中缓存磁盘块，减少磁盘 I/O：

- **缓存组织**：如何管理缓存的磁盘块？链表？哈希表？
- **淘汰策略**：缓存满时驱逐哪个块？LRU？FIFO？Clock？
- **并发控制**：多个进程可能同时访问同一个磁盘块。缓存需要提供同步机制（如每个 buffer 有一个 sleeplock）。
- **脏页回写**：被修改但尚未写入磁盘的块何时回写？

你需要回答的问题：
- 你的缓存最多缓存多少个块？这如何影响内存占用？
- 缓存的一致性如何保证？（进程 A 修改了块 X 的缓存副本，进程 B 随后读取块 X 时读到的是最新版本吗？）

### 维度 3：磁盘布局

磁盘上数据如何组织？

典型的布局：

```text
[ superblock | inode 区 | 数据区 | 日志区（可选）]
```

- **Superblock**：记录文件系统的全局信息（块大小、inode 数量、空闲块位图等）
- **Inode 区**：存储文件的元数据（大小、类型、数据块指针）
- **数据区**：存储文件和目录的实际数据
- **日志区**（可选）：用于崩溃一致性（见维度 5）

你需要决定：
- **Inode 结构**：inode 是固定大小还是可变大小？它包含哪些字段？数据块指针如何组织？（直接指针 + 间接指针？extent？）
- **目录结构**：目录如何存储？（目录项是一系列 (name, inode_no) 对的列表？）
- **空闲空间管理**：如何追踪哪些块是空闲的？（位图？空闲链表？）

你需要回答的问题：
- 你的文件系统的最大文件大小是多少？（受限于 inode 的数据块指针数量）
- 你的文件系统的最大文件数量是多少？（受限于 inode 区大小）

### 维度 4：文件系统操作

文件系统需要支持的核心操作：

- **inode 管理**：分配/释放 inode、读/写 inode 内容
- **块管理**：分配/释放数据块
- **目录操作**：创建/删除/查找目录项
- **路径解析**：根据路径名（如 `/home/user/foo.txt`）找到对应的 inode

你需要回答的问题：
- 路径解析如何处理符号链接？（如果支持）如何处理 `.` 和 `..`？
- 目录操作是原子的吗？（创建文件的同时是否可能被另一个进程看到不完整的目录状态？）

### 维度 5：一致性与崩溃安全

如果一个操作涉及多个磁盘块的修改（如创建文件：需要修改 inode 块、目录块、空闲块位图），如果在修改过程中系统崩溃，文件系统可能处于不一致状态。

常见的解决方案：

- **不处理**（教学 OS 中最简单）：接受崩溃后文件系统可能损坏。
- **Write-ahead Logging（日志）**：先将修改写入日志，日志提交后再写入实际位置。崩溃后重放日志恢复。
- **Soft Updates**：通过精心安排写入顺序来保证一致性。
- **fsck**：启动时扫描整个文件系统，修复不一致。

你需要回答的问题（写入 ADR）：
- 你的文件系统对崩溃安全的保证级别是什么？
- 如果选择了日志，日志是物理日志（记录被修改的块内容）还是逻辑日志（记录操作）？

### 维度 6：VFS 概念

虽然文件系统的底层操作是阶段 6 的重点，但你需要为阶段 7 做好接口准备：

- **统一的文件接口**：不管底层是什么文件系统，上层通过统一的函数访问（如 `fs_read(inode, buf, offset, n)`）
- **设备与文件的统一**：是否将设备也当作文件处理？这决定了 `/dev/` 在阶段 7 如何实现。

### 维度 7：物理存储介质特性

不同存储介质有根本不同的物理特性——这些特性直接影响了文件系统设计中的取舍。你的 OS 可能只跑在 QEMU 的 virtio-blk 上，但理解以下介质的差异会让你知道"为什么文件系统要这样设计"。

**HDD（机械硬盘）：**
- **物理结构**：旋转盘片 + 移动磁头。数据按柱面(Cylinder)、磁头(Head)、扇区(Sector)的三维坐标定位（CHS 寻址，后被 LBA 线性扇区号取代）。
- **性能特征**：顺序读写 100-200 MB/s，随机读写 1-2 MB/s（因为磁头需要物理移动到目标磁道，寻道时间 5-10ms，旋转延迟 2-4ms）。读写速度在所有扇区上大致均匀。
- **对文件系统的启示**：FFS 把相关数据放在同一柱面组，就是为了减少寻道距离。电梯调度算法按扇区号排序请求，也是为了最小化磁头移动。如果你在阶段 8 选择了 O3（高吞吐 I/O），理解 HDD 的物理特性是设计 I/O 调度器的基础。

**SSD（固态硬盘——SATA/NVMe）：**
- **物理结构**：NAND 闪存芯片 + 控制器（含 FTL，Flash Translation Layer）。FTL 将 LBA 映射到 NAND 物理页，负责磨损均衡(Wear Leveling)和垃圾回收(Garbage Collection)。
- **性能特征**：顺序读写 500-3500 MB/s，随机读写 300-2500 MB/s。读写不对称——写比读慢（因为 NAND 写入前需要擦除整个块）。随机读写性能远优于 HDD（因为无机械部件，但 QD=1 时延迟仍受限于 NAND 读取时间 ~50-100μs）。
- **写放大(Write Amplification)**：FTL 为写入 4KB 数据可能要实际擦除并重写整个 256KB 的 NAND 块（写放大因子可达 60×）。文件系统的 TRIM/Discard 命令告知 SSD 哪些块已不再使用，帮助 FTL 减少无效数据的搬运。
- **对文件系统的启示**：在 SSD 上，"原地覆写"和"写新位置"的开销差异很大。ZFS/Btrfs 的 Copy-on-Write 在 SSD 上表现更好——因为 FTL 天然就做重映射，文件系统的 CoW 和 FTL 的映射更新可以对齐。

**NVMe SSD（PCIe 接口的固态硬盘）：**
- **物理结构**：与 SATA SSD 相同的 NAND 基础，但通过 PCIe 直连 CPU（绕过 SATA 控制器），使用 NVMe 协议替代 AHCI。
- **性能特征**：顺序读写 3000-7000 MB/s，随机读写 500-1500K IOPS。核心优势不只是带宽——延迟从 SATA 的 ~100μs 降到 ~20μs（因为省去了 AHCI 的寄存器往返），最大队列深度从 AHCI 的 32 提升到 65535。
- **对文件系统的启示**：NVMe 的巨大队列深度使得批量 I/O 提交和并行处理成为可能。Linux 的 io_uring 正是为此设计——一次 syscall 提交数百个 I/O 操作，NVMe 可以并行处理它们。

**SD 卡 / eMMC / UFS（移动/嵌入式存储）：**
- **物理结构**：与 SSD 类似的 NAND + 控制器，但封装更小、功耗更低。SD 卡通过 SPI 或 SDIO 接口通信，eMMC 通过并行总线，UFS 通过 MIPI M-PHY 串行链路（类似 SCSI over serial）。
- **性能特征**：SD 卡 10-100 MB/s，eMMC 100-400 MB/s，UFS 500-2000 MB/s。关键限制：功耗和发热约束使它们无法像 NVMe SSD 那样并行大量 NAND die。SD 卡的写入延迟可能极不稳定（因为卡内 FTL 可能在后台做 GC）。
- **对文件系统的启示**：FAT32 在 SD 卡上的流行不是偶然——它在随机写入和元数据开销方面极轻量。如果你在阶段 8 选择了 O2（极小足迹），你的文件系统可能需要同时跑在 virtio-blk 和 SD 卡上。

**你需要回答的问题：**
- 你的文件系统假设存储介质是 HDD、SSD，还是两者皆可？你的块分配策略（如尽量分配连续的块）为哪种介质优化？
- 你的 I/O 调度器是否区分不同介质的特性？（HDD 需要寻道优化，SSD 不需要但需要批量提交）

### 维度 8：存储接口与协议

存储设备通过各种接口协议与 CPU 通信。你的 OS 需要为至少一种协议编写驱动。理解不同协议的差异能帮你判断"我的驱动应该有多复杂"。

**AHCI（Advanced Host Controller Interface —— SATA 的控制器接口）：**
- **基本模型**：主机通过系统内存中的"命令表"(Command Table)和"命令头"(Command Header)向设备提交命令。设备通过中断或主机轮询通知完成。
- **核心数据结构**：端口寄存器（Port Registers，MMIO 映射）→ 命令列表基址(PXCLB/PXFB) → 命令头数组（每个 slot 一个命令头）→ 命令表（含 FIS 和 PRDT）。FIS（Frame Information Structure）是主机与设备间传递的命令/数据块。PRDT（Physical Region Descriptor Table）描述 DMA 数据的物理地址和大小。
- **NCQ（Native Command Queuing）**：设备可同时处理最多 32 个命令，并按最优顺序执行（不是提交顺序）——类似 HDD 的电梯调度，但由设备硬件完成。
- **在 QEMU 中**：`-device ahci` 提供 AHCI 控制器，`-device ide-hd` 挂载磁盘。QEMU 的 AHCI 模拟比 virtio-blk 更接近真实硬件，但性能较低。

**NVMe（Non-Volatile Memory Express —— PCIe SSD 的控制器接口）：**
- **基本模型**：主机和控制器各维护一对环形队列——Submission Queue (SQ) 和 Completion Queue (CQ)。主机向 SQ 写入命令，控制器执行后向 CQ 写入完成通知。一对 SQ/CQ 称为一个 Queue Pair。每个 CPU 核心可以有独立的 Queue Pair（避免锁竞争）。
- **核心数据结构**：SQ Entry（64 字节固定大小，含 opcode、PRP/SGL 地址、namespace ID、逻辑块地址）、CQ Entry（16 字节，含 SQ ID、command ID、status）。PRP（Physical Region Page）或 SGL（Scatter-Gather List）描述数据 buffer 的物理地址。
- **Admin vs I/O Queues**：Admin Queue Pair 用于控制器管理命令（identify、create/delete I/O queues、get features），I/O Queue Pairs 用于实际数据读写。
- **在 QEMU 中**：`-device nvme` 提供 NVMe 控制器。NVMe 驱动的复杂度高于 virtio-blk 但低于 AHCI——因为 NVMe 的寄存器接口设计更现代、更简洁。

**virtio-blk（虚拟化块设备 —— QEMU/KVM 环境）：**
- **基本模型**：通过 virtio 描述符环（descriptor ring）提交请求。每个请求是一个描述符链——头描述符包含请求类型（读/写/flush/discard）和扇区号，后续描述符指向数据 buffer。设备完成时在 used ring 中写入完成状态。
- **优势**：接口极简（对比 AHCI 和 NVMe），性能优秀（因为 virtio 设计为半虚拟化——Guest 知道自己在虚拟机中，可以跳过真实硬件的兼容层）。
- **建议**：如果你的 OS 只跑在 QEMU 上，virtio-blk 是最佳起点。实现后，NVMe 是最自然的下一步（因为它也是队列模型）。
- **在 QEMU 中**：`-device virtio-blk-device,drive=drive0`。

**eMMC / UFS（嵌入式存储接口）：**
- **eMMC**：通过并行数据总线（1/4/8 bit）与主机通信。使用命令/响应协议（类似 SD 卡）。命令 48 位（含命令号、参数、CRC）。数据在 DAT 线上双向传输。
- **UFS**：eMMC 的继任者。使用 MIPI M-PHY 串行链路 + SCSI 命令集（通过 UFS Transport Protocol 封装）。支持多 lane、全双工、命令队列。性能接近 NVMe。
- **在 QEMU 中**：有限的 eMMC 支持。大多数教学 OS 不需要实现。

**你需要回答的问题：**
- 你的块设备驱动暴露给文件系统层的接口是什么？`block_read(dev, sector, buf)` / `block_write(dev, sector, buf)`？
- 你的驱动是同步还是异步的？同步驱动在 I/O 完成前阻塞调用线程（实现简单）；异步驱动支持提交后立即返回，通过回调/中断通知完成（性能更好，适合 O3 高吞吐 I/O）。
- 是否支持多队列？virio-blk 从 1.0 开始支持 multi-queue，NVMe 原生支持。多队列在多核场景下消除锁竞争。

### 维度 9：磁盘分区表

在文件系统之前，磁盘上有一层更底层的结构——分区表。它告诉 OS："这块磁盘被分成了几块区域？每块从哪个扇区开始、到哪个扇区结束？哪块是可引导的？"

**MBR（Master Boot Record —— 传统 BIOS 分区表）：**
- **位置**：磁盘的第一个扇区（LBA 0）。前 446 字节是引导代码（BIOS 加载并执行它），接下来 64 字节是 4 个分区表项（每项 16 字节），最后 2 字节是魔数 `0x55AA`。
- **分区表项结构（每项 16 字节）**：
  ```
  Offset | Size | 字段
  -------|------|------
   0     |  1   | 引导标志 (0x80 = 可引导, 0x00 = 不可引导)
   1     |  3   | 起始 CHS 地址
   4     |  1   | 分区类型 (0x83 = Linux, 0x0C = FAT32 LBA, 0xEE = GPT 保护)
   5     |  3   | 结束 CHS 地址
   8     |  4   | 起始 LBA（逻辑扇区号）
   12    |  4   | 扇区总数
  ```
- **限制**：最多 4 个主分区。通过"扩展分区"(Extended Partition)可以突破 4 个限制——扩展分区是一个特殊的主分区，内部包含一个链表结构的逻辑分区。最大磁盘 2TB（因为扇区总数是 32 位 × 512 字节/扇区 = 2TB）。
- **在你的 OS 中**：MBR 解析极其简单——读 LBA 0，检查魔数 `0x55AA`，遍历 4 个分区表项，记录每个有效分区（类型 ≠ 0x00）的起始 LBA 和大小。

**GPT（GUID Partition Table —— UEFI 分区表）：**
- **位置**：LBA 1 开始存储 GPT Header，LBA 2-33 存储 128 个分区表项（每个 128 字节，典型 LBA 2-33 共 32 扇区可容纳 128 个分区）。LBA -1 到 -33 存储分区表项的备份副本。
- **保护性 MBR**：LBA 0 仍包含一个 MBR 结构，但其唯一的目的是标记整块磁盘为类型 `0xEE`（GPT 保护），防止不认识 GPT 的工具错误地修改磁盘。
- **GPT Header 结构（LBA 1）**：
  ```
  Offset | Size | 字段
  -------|------|------
   0     |  8   | 签名 ("EFI PART")
   8     |  4   | 修订版本
   12    |  4   | Header 大小（通常 92 字节）
   16    |  4   | Header CRC32
   24    |  8   | 当前 LBA（即 1）
   32    |  8   | 备份 Header LBA（即磁盘最后一个 LBA）
   40    |  8   | 第一个可用 LBA
   48    |  8   | 最后一个可用 LBA
   56    |  16  | 磁盘 GUID
   72    |  8   | 分区表项数组起始 LBA（通常 2）
   80    |  4   | 分区表项数量
   84    |  4   | 每个分区表项大小（通常 128）
   88    |  4   | 分区表项数组 CRC32
  ```
- **分区表项结构（每项 128 字节）**：
  ```
  Offset | Size | 字段
  -------|------|------
   0     | 16   | 分区类型 GUID
   16    | 16   | 分区唯一 GUID
   32    |  8   | 起始 LBA
   40    |  8   | 结束 LBA（包含）
   48    |  8   | 属性标志
   56    | 72   | 分区名称（UTF-16LE）
  ```
- **优势**：支持最大 9.4 ZB 的磁盘（64 位 LBA），最多 128 个分区（无主/扩展/逻辑之分），CRC32 校验（检测损坏），备份副本（容错）。
- **在你的 OS 中**：GPT 解析比 MBR 复杂——需要读取 LBA 1 的 GPT Header（验证签名和 CRC），然后读取分区表项数组（验证 CRC），遍历每个有效的分区（类型 GUID ≠ 全零）。如果主 GPT 损坏，尝试从磁盘末尾读取备份 GPT。

**MBR vs GPT 选择指南：**

| 维度 | MBR | GPT |
|------|:---:|:---:|
| 最大磁盘容量 | 2 TB | 9.4 ZB |
| 最大分区数 | 4（主分区） | 128（典型） |
| 校验 | 无（仅魔数） | CRC32（header + entries） |
| 备份 | 无 | 磁盘末尾有备份副本 |
| 引导方式 | BIOS/Legacy | UEFI |
| 解析复杂度 | ★☆☆☆☆ | ★★★☆☆ |
| 适用场景 | 教学/小磁盘 | 现代系统 |

**在你的 OS 中实现：** 建议从 MBR 开始——它是教学场景中最简单的分区方案。只需解析一个扇区就能发现所有分区。GPT 作为进阶（如果你的 OS 支持 UEFI 引导或多于 4 个分区）。

**你需要回答的问题：**
- 你的 OS 支持哪种分区表？如果只支持一种，是什么？如果两者都支持，如何区分？（答案：读 LBA 0，检查 MBR 分区类型 —— 如果有一个类型 `0xEE`，则该磁盘使用 GPT。）
- 你的 OS 如何处理无分区表的"裸"磁盘？（如直接将文件系统写在磁盘上，无分区表。）
- 分区发现后，如何将分区信息传递给文件系统层？（每个分区作为一个独立的块设备？还是文件系统层理解分区概念？）

## 6.6 常见陷阱

1. **Buffer cache 和磁盘内容不一致**：修改了 buffer 但忘记标记为脏，导致写入丢失。
2. **日志 incomplete write**：日志写入了一半时崩溃，恢复逻辑需要能处理不完整的日志。
3. **Inode 引用计数泄漏**：分配了 inode 但在错误路径上忘记释放。
4. **路径解析的 TOCTOU 问题**：先检查权限再打开文件，但中间文件被替换。在教学 OS 中可能不需要处理，但要知道存在这个问题。
5. **磁盘扇区与文件系统块大小不一致**：磁盘扇区通常是 512 字节，文件系统块可能是 1024 或 4096 字节。需要处理部分块写入。
6. **Bread 返回的缓冲区忘记释放（brelse）**：每个 `bread()` 必须对应一个 `brelse()`。如果忘记 release，buffer cache 会被慢慢耗尽——症状是文件系统运行一段时间后所有 `bread()` 都返回 NULL（"磁盘"满了但实际上有空间——只是 buffer cache 满了）。
7. **/ 目录的 inode 号假设为 1**：xv6 中根目录的 inode 编号是 1。如果你用自己的布局，确保 `namei("/")` 返回的是真正的根 inode，而不是碰巧编号为 1 的某个 inode。

### 6.6.1 从裸机看存储：STM32 的 Flash 操作 vs OS 文件系统

如果你在 STM32 上操作过 Flash 存储，你可能经历过这些：

```c
// STM32 裸机：向内部 Flash 写入数据
HAL_FLASH_Unlock();
HAL_FLASH_Program(FLASH_TYPEPROGRAM_WORD, 0x08010000, my_data);
HAL_FLASH_Lock();

// 或者操作 SD 卡
// 发送 CMD17 (READ_SINGLE_BLOCK) → 等待 SPI 响应 → 读 512 字节 → CRC 校验
```

在裸机上，"写数据到存储"意味着：
1. 你知道数据要写到哪个物理地址（Flash 的第几个扇区、SD 卡的第几个 block）
2. 你手动遵守硬件的时序协议（Flash 需要先擦除再写入、SD 卡需要先发命令再等响应）
3. 你需要处理写入失败（Flash 磨损、SD 卡拔出）
4. 不同存储介质之间没有统一接口——Flash 用 `HAL_FLASH_Program`，SD 卡用 SPI 命令，EEPROM 用 I²C

文件系统在这一切之上加了三层抽象：

```
裸机：     你的代码 → Flash 物理地址
OS：       你的代码 → open/read/write → VFS → 文件系统 → Buffer Cache → 块设备驱动 → 磁盘
```

**每一层解决一个裸机上的痛点：**
- **Buffer Cache**：裸机上每次读 SD 卡都要发 SPI 命令——慢。Buffer Cache 把最近读过的磁盘块保存在内存中，重复读直接命中缓存。
- **文件系统层（inode / 目录 / 超级块）**：裸机上你需要记住"数据在 Flash 的 0x08010000"。文件系统用目录树和文件名来定位数据——你不需要知道它在磁盘的哪个扇区。
- **VFS 层**：裸机上 Flash、SD 卡、EEPROM 各有各的 API。VFS 统一成 `read/write`——你不需要知道底层是什么存储介质。

**对零基础自学者的启示**：如果你觉得文件系统"概念太多"（inode、目录项、超级块、日志……），回到裸机的痛点上想——没有文件系统的时候，你是怎么在 Flash 上存东西的？你手动记"配置数据在 Flash 的第 3 个扇区"——inode 就是自动帮你做这件事的。断电后数据怎么恢复？文件系统日志就是你裸机程序里的"把关键数据写两份"的工程化版本。

## 6.7 ⚡ 挑战：写前日志、mmap 与块设备抽象

### 挑战 A：实现 Write-Ahead Logging (WAL)

即使你的 OS 只跑在 QEMU 上，模拟崩溃恢复也是对文件系统正确性的终极测试。

**WAL 的基本流程**：
1. `begin_op()` — 在内存中积累本次事务的所有修改
2. `log_write()` — 将修改的块先写入日志区（而非原地覆写），记录块的最终内容
3. `end_op()` — 提交日志头（commit record），标记此事务完整
4. 后台将日志中的块逐个写回磁盘的实际位置（checkpoint）
5. 崩溃恢复时：扫描日志区，重放所有已提交的事务，忽略未完成的事务

**易错点**：
- 日志提交必须是原子的：commit record 写入完成前崩溃 = 事务丢弃，写入后崩溃 = 事务重放
- 日志空间回收：已写回磁盘的日志块可以标记为可重用——但必须在确认写回完成后
- 多次崩溃：恢复过程中再次崩溃应仍然是安全的（幂等恢复）

**验证**：在 `end_op()` 和 checkpoint 之间插入 `while(1){}` 模拟崩溃。重启后验证：要么事务的修改完全生效，要么完全不存在——不允许"改了一半"的状态。

### 挑战 B：实现 mmap 的基础版本

`mmap` 将文件映射到进程的虚拟地址空间——读写内存就等于读写文件。这是现代 OS 中最重要的 I/O 原语之一。

**简化版 mmap 的实现思路**：
1. 在进程的地址空间中预留一段虚拟地址
2. 不立即加载文件内容——设置页表条目为无效，但记录"这个虚拟地址对应文件的哪个偏移"
3. 进程访问映射区域时触发 page fault
4. page fault handler：从磁盘读取对应的文件块到物理页，建立映射
5. `munmap` 时：将脏页写回文件，释放映射

**教学价值**：mmap 是"内存管理"和"文件系统"的交叉点——它迫使你重新审视两者的接口。你的页错误 handler 现在需要知道文件系统。

### 挑战 C：通用块设备抽象

不要让你的文件系统直接依赖 virtio-blk。定义一个通用块设备接口：

```text
block_read(dev, block_no, buf) → 错误码
block_write(dev, block_no, buf) → 错误码
block_register(dev, ops)         // 注册块设备驱动
```

**这个抽象的长期价值**：当你在阶段 8 实现 USB Mass Storage 驱动时，你只需要调用 `block_register()` 注册 U 盘为新的块设备——文件系统层不需要修改。

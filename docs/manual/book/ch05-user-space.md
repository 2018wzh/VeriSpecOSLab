# 第 5 章：用户空间 — 隔离、抽象与执行

## 5.1 本章要解决的问题

你的内核现在能在 S-mode 运行、管理内存、响应中断。但用户程序还无法运行——它们被限制在更低的 U-mode 特权级，需要通过精心设计的接口与内核交互。

本章要解决的是一组紧密耦合的问题：

- **特权切换**：用户程序在 U-mode 运行，如何安全地进入内核（S-mode）请求服务？
- **Trap 处理**：当用户程序触发异常（如非法指令、页错误）或中断发生时，内核如何响应？
- **进程抽象**：如何表示一个"正在运行的程序"？它的状态是什么？如何创建、销毁和切换？
- **Syscall 接口**：用户程序如何表达它对内核服务的需求？
- **程序加载**：如何把磁盘上的可执行文件变成内存中的运行进程？
- **调度**：多个进程如何共享有限的 CPU？

这些问题是互锁的——你不能在没有 trap 机制的情况下实现 syscall，不能在没有进程抽象的情况下实现调度。因此合并为一个统一的阶段。

## 5.2 Unix 进程模型的诞生——fork 和 exec 为什么长这样

理解本章设计的入口，是理解 Unix 的进程模型从何而来。

但在这之前，先让我们澄清一个问题：**"进程"这个概念是从哪来的？** Unix 把进程做得极其优雅，但进程本身不是 Unix 发明的。

### "进程"概念的发明：从 THE 到 RC 4000

**Dijkstra 的 THE 操作系统（1968）——第一次把"进程"变成精确的概念。** Edsger Dijkstra 在 1965-1968 年间领导开发的 THE（Technische Hogeschool Eindhoven）操作系统，虽然只是一个大学项目，但它的影响远超其规模。Dijkstra 在这篇后来获得图灵奖的论文中，第一次把"顺序进程"（sequential process）定义为一个精确的计算模型——一个进程是一个顺序执行的指令流，多个进程在逻辑上同时推进，进程之间通过显式的同步机制（信号量——也是 Dijkstra 在这个项目中发明的）协调。

THE 的另一个关键贡献是**分层设计**——把操作系统分成功能上独立的几层（第 0 层：CPU 调度；第 1 层：内存管理；第 2 层：控制台 I/O；第 3 层：文件系统；第 4 层：用户程序）。每一层只能调用下层提供的服务，不能跨层调用。这种结构化的设计方法在今天看起来理所当然——但 1968 年之前，没有操作系统是按照"分层的、可证明正确的"的原则设计的。

> **原始文献：** E. W. Dijkstra, "The Structure of the 'THE'-Multiprogramming System," *Communications of the ACM*, vol. 11, no. 5, pp. 341-346, May 1968. 这篇论文只有 6 页——但其中包含了顺序进程、信号量同步、分层设计、不变量验证四个改变了操作系统设计方法论的概念。Dijkstra 就是在这篇论文中写出了那句著名的话："测试可以证明 bug 的存在，但不能证明 bug 的不存在。"

**RC 4000 多道程序系统（1969）——进程通信的第一次系统化尝试。** 丹麦计算机科学家 Per Brinch Hansen 在为 RC 4000 计算机设计操作系统时，面临一个此前没人系统解决过的问题：**如果多个进程是独立的、隔离的实体，它们怎么互相通信？** 他的答案——也是操作系统历史上第一次——是把"消息传递"（message passing）作为进程间通信的唯一原语。每个进程有一个消息队列，进程通过 `send` 和 `receive` 操作交换数据。内核只提供发送和接收的机制，不参与消息的内容解释。

Brinch Hansen 的 RC 4000 核（nucleus）是现代微内核 IPC 的直系祖先。seL4 的 Endpoint、QNX 的消息传递、甚至 Unix 的 pipe——所有这些 IPC 机制的根，都可以追溯到 RC 4000 的 "send/receive" 模型。

> **原始文献：** P. Brinch Hansen, "The Nucleus of a Multiprogramming System," *Communications of the ACM*, vol. 13, no. 4, pp. 238-241, April 1970. Brinch Hansen 后来在 1973 年出版了 *Operating System Principles*——第一本系统阐述进程同步、IPC 和调度算法设计原则的教科书。

### fork 的起源：一个"本该是临时的"方案

1971 年，Unix 的进程创建方式不是 fork。它使用 `fork` 的一个前身——创建一个子进程，并指定子进程要运行的程序。但 PDP-7 的硬件限制迫使 Ken Thompson 和 Dennis Ritchie 做了一个简化："复制当前进程，然后让子进程自己去替换程序。"他们本以为这是一个临时方案，但后来发现这个两步模型（fork + exec）有出乎意料的好处。

fork 之后、exec 之前的那一瞬间——子进程是父进程的完整复制品，但可以修改自己的文件描述符、环境变量、工作目录——然后再 exec 到新程序。这让 Shell 能够做 I/O 重定向：`fork` → 子进程关掉自己的 fd 1 → 打开目标文件（它自动成为新的 fd 1）→ `exec` 运行目标程序。目标程序往 fd 1 写数据，数据就进了文件。它不需要知道"自己正在被重定向"。

这个设计的美妙之处：**程序不需要知道自己在什么上下文中运行。** Shell 通过 fork+exec 之间的窗口，悄悄修改了子进程的环境，而子程序对此毫无感知。这是 Unix 组合性的基石。

### fork 的代价

但 fork 的设计在今天看来也背负了严重的技术债务。

fork 需要完整复制父进程的地址空间——在 1971 年这很简单（PDP-7 只有 8 KB 内存），在 2020 年代这是巨大的开销。Copy-on-Write（CoW）缓解了大部分问题——fork 时不复制物理页，只标记为只读，父子进程共享；谁先写入，MMU 触发 page fault 时才复制那一页。但 CoW 也带来了自己的复杂度——引用计数、page fault handler 中的 CoW 逻辑、多核下的并发控制。

后来的系统（如 Windows NT、Plan 9 的 rfork、Linux 的 clone/posix_spawn）都在尝试摆脱 fork 的束缚。Windows 的 `CreateProcess` 是一步创建——不复制，直接创建一个新进程并加载程序。这避免了 fork 的大多数开销和复杂性，但失去了 Unix 那种"fork 和 exec 之间的窗口可以自由修改子进程环境"的灵活性。

### 对你这门课的意义

你可以选择 fork+exec 两步模型（Unix 传统），也可以选择 spawn 一步模型（更现代）。fork+exec 的教学价值在于它展现了 Unix 设计哲学——"把复杂的事情拆成两个简单步骤，让中间状态可被利用"。spawn 的教学价值在于它更干净——不需要 CoW、不需要复制地址空间、进程创建的语义更集中。

两种选择都合理。你的 ADR 需要解释为什么选这种。

## 5.3 设计维度

### 子阶段 A：特权级与 Trap

#### 维度 A1：Trap 向量与分发

当 CPU 在 U-mode 遇到 trap（异常或中断）时，硬件自动执行一套固定的流程：切换到 S-mode，把 trap 发生时的 PC 保存到 `sepc`，把 trap 原因写入 `scause`，把相关信息写入 `stval`，然后把 PC 设为 `stvec` 指向的地址。

你需要决定 trap vector 的组织方式。`stvec` 的低位决定模式：直接模式（`stvec` 直接指向分发函数，所有 trap 都跳到同一个地址）或向量模式（`stvec` 指向跳转表，按 `scause` 索引）。教学场景中直接模式最简单——所有 trap 进入同一个入口，在 C 代码中根据 `scause` 做 `switch-case`。

#### 维度 A2：Trap 处理路径的四阶段

trap 处理分为四个阶段，每个阶段的失败模式不同。

第一阶段是**入口**——汇编代码，保存全部通用寄存器到 trapframe 结构，切换栈指针到内核栈，设置内核执行环境。为什么必须用汇编？因为 C 函数调用约定会破坏 `a0-a7` 和 `ra` 寄存器——而这些正是用户程序的执行上下文。你必须在 C 代码执行之前就把它们保存到安全的地方。

第二阶段是**分发**——C 代码根据 `scause` 判断 trap 类型并调用对应的 handler。这一步的难点在于区分类别：来自 U-mode 的 ecall（syscall）→ 调用 syscall 分发表；来自 U-mode 的 page fault → 检查地址合法性，决定是修复映射还是终止进程；来自 S-mode 的异常 → 这是内核 bug，通常直接 panic；timer 中断 → 可能触发进程调度。

第三阶段是**处理**——各个 handler 执行具体逻辑。关键是 handler 不能假设自己运行在"安全"的环境中——用户程序可能传入了恶意参数，handler 必须验证所有来自用户态的输入。

第四阶段是**返回**——汇编代码从 trapframe 恢复寄存器，执行 `sret` 指令。`sret` 将 `sepc` 恢复到 PC，将 `sstatus.SPP` 恢复到特权级，完成从 S-mode 到 U-mode 的跳转。

#### 维度 A3：Trampoline 页——RISC-V 特有的挑战

RISC-V 的特权架构在 trap 入口产生了一个微妙问题：进入 trap 时，CPU 已经切换到了 S-mode，页表还是用户页表。你需要一条"在用户页表和内核页表里都映射到相同虚拟地址"的代码路径——这就是 trampoline 页。

Trampoline 页映射在两个页表的同一个虚拟地址上（比如都映射在 `0x3FFFFF000`——虚拟地址空间的最高页）。它包含 trap 入口和返回的代码。为什么这个设计是必需的？因为 trap 入口需要：使用用户页表从 `sscratch` 加载 trapframe 地址；切换到内核页表（写 `satp`）；然后跳转到内核空间的其他代码。如果 trampoline 代码不在两个页表共享的位置，切换 `satp` 后下一条指令的 PC 就会落在"新页表中不存在的虚拟地址"上——立即崩溃。

Trampoline 页必须标记为 U=0（只允许内核访问），防止用户程序修改它。这是你内核中最核心的安全边界之一。

#### 维度 A4：熔断漏洞与页表隔离——为什么“同一页表”从默认选项变成了安全漏洞

Trampoline 页的存在，本身就是一个设计妥协的证据——为什么要在两个页表里共享一页代码？因为 RISC-V 的 trap 入口需要同时看到用户映射和内核映射。但如果 CPU 本身有一个更根本的安全缺陷，连“U=0”这个权限位都不能信任——你的整个隔离模型会怎样？

这正是 2018 年 1 月发生的事情。

**Meltdown 的核心机制。** 现代 CPU 是乱序执行的——遇到一条指令需要等慢速内存时，CPU 不会闲着，而是继续推测执行后面的指令。关键问题在于：Intel（以及部分 ARM 和 IBM POWER）CPU 在推测执行时**不检查页表的 U/S 权限位**。权限检查只在最后“提交”（commit）阶段进行——如果发现违规，CPU 回滚所有架构可见的状态（寄存器、内存）。但推测执行期间加载到 cache 的数据**不会被回滚**。

攻击者利用这一点：
1. 用户程序申请一个数组（在用户空间），大小刚好覆盖多个 cache line
2. 用 `ecall` 或等价手段训练分支预测器
3. 执行一条访问内核地址的 load 指令——例如 `mov rax, [kernel_addr]`
4. CPU 在推测执行中：忽略 U=0 权限位 → 从 L1 cache 读取内核数据 → 用这个数据作为索引访问用户数组的某个元素 → 该元素所在 cache line 被加载到 cache
5. 权限检查最终失败，CPU 回滚。但用户数组的特定 cache line 已经留在 cache 里了
6. 用户程序遍历自己的数组，测量每个元素的访问延迟——访问快的是 cache hit，慢的是 cache miss。cache hit 的那个索引 = 被泄漏的内核数据的值

**后果。** 任何运行在 Intel CPU 上的操作系统——Linux、Windows、macOS——其全部内核内存（物理内存、进程表、文件缓存、加密密钥）都可以被任意用户态程序以约 500 KB/s 的速度逐字节读出。修复这个漏洞不能靠微码更新——它的根源是 Intel CPU 的推测执行设计缺陷。

**KPTI（Kernel Page Table Isolation）——软件层面的止血方案。** Linux 社区的应对措施是 KPTI（原名 KAISER）：每个进程维护两份页表——一份“用户页表”（只有用户映射 + trampoline + 极少数内核入口代码），一份“内核页表”（完整的用户映射 + 完整的内核映射）。进程在用户态运行时使用用户页表——这张表里几乎没有内核映射，所以即使 CPU 推测执行越界，也没有内核数据可泄漏。进入内核时切换到内核页表。这个页表切换就是你的 trampoline 页中的 `satp` 写入操作。

**性能代价。** 每次 syscall 都要写 `satp`（RISC-V）或 `CR3`（x86），并伴随 TLB 刷新。2018 年的测量显示性能下降 5-30%——取决于工作负载的 syscall 密度。Intel 在后续微架构（Cascade Lake 及之后）中加入了硬件缓解，但软件层面的页表隔离仍然是纵深防御的标准配置。

**RISC-V 的情况。** RISC-V ISA 在设计时吸取了 Meltdown 的教训——特权规范明确禁止在推测执行中绕过权限检查。当前的主流 RISC-V 实现（SiFive、香山等）不受 Meltdown 类攻击影响。但这不意味着你可以忽略页表隔离。

**Spectre——比 Meltdown 更难修复的幽灵。** 如果说 Meltdown 是某个 CPU 厂商的实现错误，Spectre 就是一个**方法论级别的漏洞**——它攻击的不是权限检查的遗漏，而是分支预测器本身。

现代 CPU 的分支预测器本质上是一个模式匹配器：你给它看足够多的历史，它就学会猜测下一条分支往哪走。Spectre 的核心思路是：**先训练分支预测器，让它“学会”一个错误的方向，然后诱导 CPU 在推测执行中访问不该访问的数据。**

Spectre 有两个经典变体：

**变体 1（Bounds Check Bypass，边界检查绕过）。** 攻击者构造一个数组访问，边界检查（`if (index < size)`）在正常情况下会阻止越界。但攻击者先反复用合法 index 调用这段代码——分支预测器学会了“这个 if 总是成立的”。然后突然传入一个越界的 index。CPU 在等待边界检查结果时，预测“if 成立”并推测执行了越界访问。越界读到的数据被用于后续的 cache 侧信道传输。等边界检查完成、CPU 发现违规并回滚时，数据已经通过 cache 泄漏了。

关键区别：Spectre v1 **不需要访问内核内存**。受害者可以是任何有合法访问权限的代码——一个 JavaScript 沙箱里的 `ArrayBuffer` 边界检查，一个内核中验证用户传入的 index 是否在数组范围内的检查。攻击者在同一个地址空间内、同一个特权级下就能完成攻击。页表隔离对此完全无效——因为攻击者和受害者共享同一套页表。

**变体 2（Branch Target Injection，分支目标注入）。** 攻击者污染间接分支预测器（BTB），让 CPU 在遇到 `jalr` 或 `ret` 指令时，推测执行跳转到攻击者选定的地址。这个地址可以是一个“gadget”——一段现存代码，其副作用（如 `load [rdi]`）可以将敏感数据带入 cache。攻击者在自己的地址空间中反复训练 BTB，使得目标进程的间接跳转被预测到 gadget 地址。由于 BTB 在许多 CPU 上是跨进程共享的（按虚拟地址索引），攻击者可以在自己的进程中用相同的虚拟地址“毒化”预测器，然后等待受害者进程执行到那条指令。

Spectre v2 同样不依赖内核映射的存在。它攻击的是跨进程的预测器状态共享——这是 CPU 为了性能做出的另一个设计选择。

**为什么 Spectre 比 Meltdown 更难修复。** Meltdown 有一个明确的根因（推测执行不检查权限位）和一个相对直接的缓解方案（页表隔离）。Spectre 的根因是“分支预测器可以被跨进程或跨特权级的状态所影响”——这是分支预测的基础工作机制，不是 bug。修复方案（如 Retpoline、IBRS、STIBP）都伴随着显著的性能代价，且每种 CPU 微架构需要不同的缓解策略。2026 年的今天，Spectre v1 仍然主要通过编译器插入 `lfence` 屏障来缓解——这要求开发者识别出哪些边界检查是安全敏感的，手工或通过编译器静态分析插入屏障。

**对你 OS 的启示。** 你正在实现一个操作系统。Meltdown 和 Spectre 给你上了同一课的两个侧面：
- Meltdown 教给你：**地址空间的隔离不能只依赖权限位。** 如果你的内核和用户共享页表，硬件可能在推测执行中偷偷越界。页表隔离是你的朋友的。
- Spectre 教给你：**地址空间的隔离不能只依赖页表。** 即使在同一个地址空间内，你的边界检查也可能在推测执行中被绕过。`copyin`/`copyout` 的参数验证、syscall 中的 index 范围检查——这些不仅仅是“防御恶意用户程序”的措施，它们在推测执行时代也是“防御 CPU 本身”的措施。

### 子阶段 B：进程抽象

#### 维度 B1：进程表示

你如何表示一个进程？

至少需要包含的信息：
- **进程标识**：PID 或其他唯一标识
- **CPU 状态**：进程切换时需要保存的寄存器
- **内存状态**：指向该进程页表的引用
- **执行状态**：running / runnable / sleeping / zombie 等
- **资源句柄**：进程打开的文件、pipe 等资源

你需要回答的问题：
- 你的进程状态机有几种状态？状态之间的转换条件是什么？
- 进程间的关系如何管理？是否支持进程树（父子关系）？

#### 维度 B2：进程生命周期

- **创建**：如何创建一个新进程？从零创建（类似 Windows `CreateProcess`）还是复制现有进程（类似 Unix `fork`）？
- **执行**：如何让新进程运行一个新程序？替换地址空间（类似 `exec`）还是其他方式？
- **终止**：进程如何退出？它的资源如何回收？
- **等待**：父进程如何等待子进程结束？退出状态如何传递？

你需要回答的问题：
- `fork` + `exec` 的两步模型 vs `spawn` 的一步模型——你的选择是什么？
- 孤儿进程（父进程先于子进程退出）如何处理？
- 僵尸进程（子进程已退出但父进程未回收）如何处理？

#### 维度 B3：上下文切换

从一个进程切换到另一个进程，需要：
- 保存当前进程的寄存器状态
- 切换页表（`satp` 写入目标进程的页表地址）
- 恢复目标进程的寄存器状态

这个过程需要汇编实现（无法用纯 C 完成，因为涉及寄存器操作和栈切换）。

你需要回答的问题：
- 上下文切换的原子性如何保证？（通常在持有调度器锁的情况下进行）
- 哪些寄存器需要保存？哪些不需要？（调用约定决定了 callee-saved vs caller-saved）

### 子阶段 C：Syscall 与程序加载

#### 维度 C1：Syscall 分发机制

用户程序通过 `ecall` 指令触发 syscall。内核需要：
- 知道用户请求了哪个 syscall（通过 `a7` 寄存器或其他约定）
- 获取 syscall 参数（通过 `a0-a5` 寄存器或其他约定）
- 执行对应的内核函数
- 将返回值放入 `a0`

你需要回答的问题：
- 你的 syscall 分发表如何组织？数组索引？哈希表？
- 如何处理未定义的 syscall 编号？
- syscall 参数中如果有用户指针，如何安全地访问？

#### 维度 C2：用户指针安全

用户程序可能传入任意指针给 syscall。内核必须验证：
- 指针指向用户地址空间（不能指向内核地址空间）
- 指针指向的内存区域存在且可访问
- 对于写操作，目标内存区域可写

你需要回答的问题：
- 你的指针验证策略是什么？在 syscall 入口统一验证，还是在每个 syscall 实现中单独验证？
- 验证失败时返回什么错误？

#### 维度 C3：可执行格式与加载器

用户程序以什么格式存储在磁盘上？你的加载器如何将其加载到内存？

- **ELF 格式**：最通用的可执行格式。至少需要解析 ELF header 和 Program Headers，加载 LOAD 段。
- **自定义格式**：如果你选择自定义格式，需要定义明确的格式规范。
- **静态 vs 动态链接**：教学场景中通常只支持静态链接，简化加载器。

你需要回答的问题：
- 你的加载器验证 ELF 文件的完整性吗？（magic number、架构匹配、段对齐）
- 加载器如何设置新进程的栈和初始寄存器状态？
- Argv 和环境变量如何传递给新进程？

## 5.4 调度器简史——不只是"选下一个跑谁"

计算机科学家花费了几十年研究一个问题："在多个等待运行的程序中，应该先跑谁？"答案取决于你觉得什么是"公平"的。

**FCFS（First-Come-First-Served）——最简单，最不公平。** 先来的先跑，后来的等着。这是批处理时代的自然做法——程序按提交顺序一个一个跑。问题是"短作业等长作业"：一个只需要跑 1 秒的程序，排在一个需要跑 1 小时的程序后面——它等 1 小时只为了跑 1 秒。

**SJF（Shortest Job First）——最优平均等待时间，但不可实现。** 理论上，总是选最短的程序先跑，能让所有程序的平均等待时间最短。问题是你不知道一个程序要跑多久——它还没跑完你怎么知道它还要多久？SJF 只能用在你知道运行时间的场景（比如特定类型的批处理作业）。

**Round-Robin——\"每个程序都能往前推一点\"。** 分时系统引入了"时间片"的概念——每个程序跑一小段时间，然后CPU 被抢走，换下一个程序。这就是 round-robin。每个程序都能得到进展，没有一个程序被完全饿死。时间片大小是关键——太小（1ms）导致上下文切换开销占主导；太大（100ms）导致交互迟钝。通常取 10-100ms。

**MLFQ（Multi-Level Feedback Queue）——\"我对你的了解越多，我越知道怎么分配 CPU 给你\"。** 这是现代操作系统中实际使用最多的算法之一。基本思路：每个程序有一个动态优先级。如果一个程序用完了它的时间片（说明它是 CPU 密集型），降低它的优先级——让它以后获得更大但更少的调度机会。如果一个程序在时间片用完前就放弃 CPU（说明它在等 I/O，是交互型程序），保持或提升它的优先级——确保你打字时编辑器能立即响应。MLFQ 不需要程序提前声明它是"CPU 型"还是"I/O 型"——它通过观察程序的行为自己学会。

**CFS / EEVDF——Linux 的现代路线。** 2007 年，Linux 用 CFS 替换了之前的 O(1) 调度器。CFS 用红黑树维护每个进程的"虚拟运行时间"（实际运行时间除以权重），每次选择虚拟运行时间较小的进程。Linux 6.6 起默认调度逻辑转向 EEVDF，在保留公平性思想的同时，用虚拟 deadline 改善延迟控制。

### 调度策略的选择

每种调度策略代表一种不同的"公平"定义。

Round-Robin 定义公平为"每个进程获得均等时间片"。实现简单（一个循环链表），行为可预测（N 个进程在固定周期内至少运行一次）。弱点是对交互型程序不够友好——一个刚被键盘唤醒的编辑器要等前面的 CPU 密集型进程用完时间片。

MLFQ 定义公平为"根据进程的历史行为动态调整——交互型获得响应性，CPU 型获得吞吐量"。通过观察进程是否用完了时间片来自动推断进程类型。弱点是需要调优的参数多（队列数、时间片大小、优先级提升间隔）。

CFS/EEVDF 定义公平为"在任意时间窗口中获得的 CPU 比例接近权重比例"，并进一步考虑可调度实体的虚拟 deadline。弱点是需要红黑树、虚拟运行时间和 deadline 计算——代码复杂度显著高于 Round-Robin。

你的选择取决于你 ArchitectureSeed 中的设计目标。如果兼容性是首要目标，你可能需要与参考系统一致的调度行为。如果教学清晰性是首要目标，Round-Robin 的行为最可预测和验证。如果你追求性能或安全，MLFQ 和 CFS/EEVDF 提供了更丰富的调度语义。

多个进程竞争 CPU，调度器决定下一个运行谁。你需要回答的问题（写入 ADR）：
- 你选择的调度策略的设计目标是什么？（公平性？响应性？吞吐量？）
- 什么事件触发调度？（进程退出、时钟 tick、进程主动 yield、进程被阻塞？）
- 如何防止饥饿？（低优先级进程永远得不到 CPU）

## 5.5 典型设计路线（参考）

### Trap 路径参考

```text
用户态执行 → ecall / 异常 / 中断
    → 硬件切换 S-mode，跳转 stvec
    → trampoline：保存寄存器到 trapframe
    → usertrap()：分发
        ├── scause == ecall from U  → syscall()
        ├── scause == page fault    → page_fault_handler()
        ├── scause == timer intr    → yield()
        └── scause == device intr   → devintr()
    → usertrapret()：准备返回
    → trampoline：恢复寄存器 → sret
    → 用户态恢复执行
```

### 进程生命周期参考

```text
fork() → 创建子进程（复制父进程的页表和 trapframe）
exec() → 加载新程序（替换地址空间）
exit() → 进程退出（释放资源，状态变为 ZOMBIE）
wait() → 父进程回收子进程（清理 ZOMBIE 进程）
```

### Syscall 分发参考

```text
syscall() 函数：
1. 从 trapframe 读取 a7（syscall 编号）
2. 从 trapframe 读取 a0-a5（参数）
3. 在 syscalls[] 表中查找对应的处理函数
4. 调用处理函数
5. 将返回值写入 trapframe 的 a0
```

## 5.8 ⚡ 进阶调度器与 CoW Fork

### 挑战 A：实现 MLFQ（多级反馈队列）调度器

Round-Robin 实现简单但无法区分 I/O 密集型和 CPU 密集型进程。MLFQ 通过观察进程行为自动调整优先级：

1. 维护 N 个优先级队列（通常 3-4 级）。最高优先级队列时间片最短（如 1 tick），每降一级时间片翻倍
2. 新进程从最高优先级进入
3. 用完时间片的进程被降到下一级
4. 在时间片用完前主动放弃 CPU 的进程（如在等 I/O）保持在当前级别
5. 定期将所有进程提升回最高优先级（防止饥饿）

**关键不变量**：低优先级队列中的进程不会饿死——优先级提升周期（如每 100 ticks）保证每个进程最终都会回到最高优先级。

**验证**：编写调度公平性不变量检查器。对每对进程 (A, B)，在任意 1000 tick 的时间窗口内，两者的运行时间比例应接近其优先级权重比例（允许 ±20% 误差）。

### 挑战 B：实现 Copy-on-Write (CoW) Fork

基础的 `fork()` 实现完整复制父进程的地址空间——这对大进程来说是巨大的浪费。CoW fork 的实现：

1. `fork()` 时复制父进程的页表，但将所有可写页标记为只读（清除 PTE_W），设置 CoW 标志
2. 父子进程共享所有物理页，每个物理页增加一个引用计数
3. 当任一进程尝试写入 CoW 页时，触发 page fault
4. page fault handler 识别 CoW 页：分配新物理页，复制内容，更新页表为可写，减少原页引用计数
5. 当引用计数降为 0 时，物理页被释放

**易错点**：
- 引用计数的并发正确性（多核场景下 `fork()` 和 page fault 可能同时操作同一页的引用计数）
- CoW 页的识别：需要一个方式在页表条目中标记"这是一个 CoW 页"（RISC-V 的 PTE 中可以使用预留位）

**验证**：`fork()` 一个 100 页的进程。验证父子进程最初共享所有物理页（引用计数 = 2），子进程修改 10 页后只有这 10 页被复制（引用计数 = 1），其余 90 页仍共享。

### 挑战 C：多核调度器的正确性

在阶段 2 你已经处理了多核启动。阶段 5 的调度器需要扩展到多核：

- 每个 CPU 有自己的调度队列 vs 全局调度队列——前者减少锁竞争但可能导致负载不均，后者公平但锁竞争严重
- 调度锁的正确获取顺序：先锁当前 CPU 的调度队列，再锁目标 CPU 的队列——如果两个 CPU 同时尝试锁对方，死锁

**验证**：编写一个多核正确性不变量检查器——在任意时刻，同一个进程不应该在两个不同的 CPU 上同时运行。

## 5.9 常见陷阱

1. **Trapframe 保存/恢复不对称**：保存了 32 个寄存器但只恢复了 31 个——这种错误在稍后才会暴露，非常难排查。
2. **用户指针验证缺失**：用户传入内核地址作为 syscall 参数，内核盲目写入，导致内核数据被破坏。
3. **fork 复制不完整**：复制了页表但忘记了 trapframe，子进程第一次 trap 就崩溃。
4. **exec 后未刷新 TLB**：旧的页表映射残留在 TLB 中，新进程可能访问到旧进程的内存。
5. **调度器死锁**：调度器在持有调度锁的情况下调用 `swtch`，导致其他 CPU 无法获取调度锁。
6. **忘记处理孤儿进程**：父进程退出后子进程成为孤儿。如果无人回收，它们会永远占用进程表条目。
7. **`sret` 后 CPU 卡在 U-mode 无法再进入 S-mode**：如果 `sstatus.SPP` 被错误设为 S-mode（而非 U-mode），`sret` 后 CPU 仍在 S-mode——但 `stvec` 的 trap 处理代码是为"从 U-mode 进入"设计的。**验证：在 `sret` 前检查 `sstatus.SPP` 的值为 0（U-mode）。**
8. **用户栈没有正确映射**：你为进程分配了栈空间、在页表中建立了映射——但忘了把栈的初始 SP 写进 trapframe。用户程序第一条 `call` 指令就炸——因为 SP 指向了未映射的地址或者随机值。
9. **Syscall 返回值没有写回 trapframe**：处理完 syscall 后，返回值放在了 C 函数的 `return` 语句里——但 trap 返回时是从 trapframe 中恢复的 `a0`，不是从 C 函数的返回值。**正确做法：`tf->a0 = syscall_result;`。**
10. **`sepc` 未在 `ecall` 处理后递增**：`ecall` 指令执行后，`sepc` 指向 `ecall` 指令本身。`sret` 后会再次执行同一条 `ecall`——无限循环。**解决：在 syscall handler 返回前 `tf->sepc += 4`（RISC-V 指令长度 4 字节）。**

## 5.10 自学导航：本阶段怎么安排时间

阶段 5 是课程中最长、最复杂的阶段。以下是一个推荐的推进顺序——不要把本章从头读到尾然后试图一次性实现所有东西。分三段推进，每段独立验证。

### 第一段：Trap 机制（预计 2-3 天）

**只做这些：** Trap 入口/返回（汇编 trampoline）、trap 分发（C dispatcher）、最小 syscall（只实现 `write` 和 `exit`）。

**验证目标：** 一个用户程序通过 `ecall` 调用 `write` syscall 在串口打印 "hello from user mode"，然后调用 `exit` 正常退出。

**跳过：** 先不做 fork/exec、调度器、进程管理器。让你的"进程"暂时只有一个——手工创建它的页表和 trapframe，手工加载它的代码到内存。

### 第二段：进程抽象（预计 2-3 天）

**在第一段基础上加入：** 进程数据结构、进程表、上下文切换（`swtch`）、调度器（最简单的 round-robin）。

**验证目标：** 两个进程被调度器交替运行——每个进程打印自己的标识字符，你看到串口输出交替的 "A" 和 "B"。

**跳过：** 先不做 fork 复制、exec 加载器。手工创建两个进程——不同的页表、不同的代码。

### 第三段：完整生命周期（预计 2-3 天）

**在前两段基础上加入：** fork（含 CoW，如果选择了）、exec（ELF loader）、wait/exit 进程回收、孤儿/僵尸进程处理。

**验证目标：** 一个 Shell 程序能通过 fork+exec 启动子进程、等待子进程结束、回收资源。

### 如果卡住了

- **Trap 入口崩溃（`sret` 后 CPU 不回到 U-mode）**：用 GDB 单步跟踪 trampoline 的每一条指令。最常见错误：`sepc` 写错了、`sstatus` 的 SPP 位设错了、trapframe 中的寄存器没正确保存。
- **Syscall 返回后用户程序行为异常**：检查 trapframe 中 `a0`（返回值）和 `sepc`（返回地址）是否正确。忘了递增 `sepc` 是最常见的原因。
- **调度器切换后第一个进程永远在跑、第二个进程从未被调度**：检查时钟中断是否正确触发、`yield()` 是否在 tick handler 中被调用。
- **fork 后子进程第一次 trap 就崩溃**：检查子进程的 trapframe 是否完整复制、子进程的页表是否包含 trampoline 映射。

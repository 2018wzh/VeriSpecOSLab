# VeriSpecOSLab 项目创新点与突出优势总结

## 一、总体定位与核心理念

VeriSpecOSLab 是一套面向操作系统课程、系统软件实践和科研训练的 **AI 辅助、规格驱动的教学实验方案**。它把传统 OS 实验的范式从"教师给框架，学生补代码，跑通测试"，换成了另一条路径：

> **学生提出架构设计 → 规格约束 AI → AI 辅助生成与验证 → 证据驱动的可审计评价**

这一思路受到 SYSSPEC 论文的直接启发。SYSSPEC 的核心主张是把系统开发的重心从低层代码编写移到高层规格设计，用验证反馈来约束 AI 的生成过程。VeriSpecOSLab 把这个思路从文件系统单一模块扩展到了完整操作系统教学实验的全链路。对学生而言，学习重点不再是"写出某段代码"，而是"定义什么是正确的、说明为什么这样设计、约束 AI 怎样实现、验证实现是否满足规格、解释架构取舍、维护和演化一个复杂系统"。这六件事合在一起，才是本方案真正想教的。

## 二、相对于传统实验的核心差异

### 2.1 高度个性化：不绑定特定内核形态

传统 OS 实验，比如 MIT 6.S081（xv6）和清华大学 rCore，几乎都限定学生实现某一种内核形态。xv6 是类 Unix 宏内核，rCore 是 Rust 微内核，学生在教师预设的框架里补全代码即可。VeriSpecOSLab 不预设任何内核形态，学生需要自己做完下面这些架构决策：

- **内核组织模型**：宏内核、微内核、混合内核、Exokernel、Library OS，或者自己定义的架构
- **执行模型**：进程、线程、任务、Fiber、Actor，以及配套的调度策略、抢占模型、优先级模型
- **保护模型**：特权级设计、地址空间隔离、权限机制（页表、Capability、Handle、命名空间权限等）
- **通信模型**：系统调用、IPC、消息传递、共享内存、事件通道，同步/异步语义、拷贝策略
- **资源模型**：FD、Handle、Capability、Port、Object 等资源抽象，生命周期管理、引用计数、所有权转移

学生用 `ArchitectureSeed`（架构种子）、递进式 `ArchitectureSlice`、`ArchitectureDecisionRecord`（ADR）和期末的 `FinalArchitectureSynthesis` 完整记录自己的设计历程。教师看到的不再是千人一面的补全代码，而是每个学生独立的设计思维。

**教师视角**：个性化设计把教师的角色从"检查代码过没过测试"推到了"审视学生的系统设计思维"。教师可以对比不同学生的架构选择，比如某个学生借鉴了 L4 的 Capability 机制但拒绝了它的 IPC 模型，由此判断学生对原理的理解到底有多深。ADR 机制还让教师能够追溯学生的决策过程，看取舍背后的理由是否站得住脚。

### 2.2 从零构建：不提供预制框架

传统实验给学生一套比较完善的操作系统源码作为起点（xv6 约 9000 行核心代码，rCore 提供完整框架），学生在上面修改和补全。本方案要求学生从零开始完成一个完整的内核：

- 起点是 `ArchitectureSeed`，学生自己定义系统目标、参考了哪些系统、借鉴了什么、修改了什么、拒绝了什么，以及为什么
- 不提供任何预制内核框架代码。学生通过 `vos agent generate` 让 Agent 根据自己写的规格生成代码骨架，再逐步完善
- 每个阶段（boot → memory → trap → process → syscall → filesystem → IPC → device）都是先完成规格设计，再进入实现

**教师视角**：从零构建逼着学生去理解每个模块"为什么存在"，而不只是"怎么实现"。传统实验里，学生完全可以在不理解页表原理的情况下，靠修改已有代码把测试跑通。但在本方案中，学生必须先写 `ModuleSpec`（模块规格）和 `OperationContract`（操作契约），把前置条件、后置条件、不变量和失败语义都交代清楚，才能进入代码生成。教师审查的是规格质量，不是代码能不能跑，这样更容易判断学生到底掌握了没有。

## 三、核心技术机制

### 3.1 规格化：重心从写代码转向做设计

这是本方案最底层的方法论变化。传统 OS 实验里，学生的大部分精力耗在手动编写和调试 C/Rust 代码上。在本方案中，规格才是项目的设计真相（source of truth），代码是实现细节。

**三层规格体系**：

| 层次 | 回答的问题 | 核心产物 |
|------|-----------|---------|
| Architecture Layer | 为什么这样设计？ | `ArchitectureSeed`、`ArchitectureSlice`、ADR |
| Module Layer | 模块管哪些状态？边界在哪？ | `ModuleSpec`（状态空间、接口族、模块不变量） |
| Operation Layer | 单个操作的前置/后置/失败语义？ | `OperationContract`（依赖、保证、并发、测试义务） |

这里有一个关键的判断：LLM 驱动开发真正需要的是操作级上下文，不是只有高层设计摘要。传统实验即便有文档，也基本停在模块总览层面。本方案要求每条核心操作都有独立的 YAML 规格，Agent 生成代码时必须绑定对应的 `OperationContract`。

规格不是写完就定型了。所有重大设计变更要走 commit-backed `SpecPatch`，先改 Spec，再改代码。`SpecPatch` 记录演化原因、影响范围、风险和必须回跑的测试，整个演化过程是可追溯的。

**教师视角**：

1. **设计文档可审查**：教师不用在大量代码里猜学生的设计意图。所有设计决策以结构化 YAML 存在，`vos spec lint` 和 `vos arch lint` 可以自动检查。
2. **设计审核成为教学环节**：教师可以在代码生成前审核学生的 `ArchitectureSlice` 和 `ModuleSpec`，在设计层面就把问题纠正过来。这比代码写完了才发现架构问题要高效得多。
3. **差异化教学**：每个学生的架构设计不同，教师的指导也可以因人而异，不用对所有人说同样的话。
4. **防止 AI 滥用**：Agent 必须绑定 Spec 条款才能写入代码，学生没办法让 AI 代写一整套系统而不理解原理。

**学生视角**：写规格这件事，一开始会觉得"多了一道工序"。但到了第三个阶段，大多数学生会发现一个反直觉的事实：花时间写清楚"这段代码应该做什么"，比直接写代码然后反复调试，总时间反而更短。因为规格帮你把模糊的想法逼成了精确的约束，而精确的约束让 Agent 生成的代码出错概率大幅降低。更重要的收获在于，规格文档本身就是最好的复习材料。期末回看自己一学期写的 ArchitectureSlice、ModuleSpec 和 SpecPatch，整个内核的设计逻辑一目了然。传统实验里，这个过程只能靠回忆和代码注释。

### 3.2 可验证化：多层次自动验证

本方案的验证体系覆盖了从开发期到评分期的完整链条，不只是"跑通几个测试用例"。

**验证层次**：

| 验证类型 | 触发时机 | 目的 |
|---------|---------|------|
| `verify public` | 每次 push 或本地开发 | 公开验证，学生可见完整结果 |
| `verify patch` | SpecPatch 应用后 | 检查设计变更的影响范围和回归 |
| `verify generated` | Agent 代码生成后 | 自动派生验证，检查生成代码是否正确 |
| `verify invariant` | 关键模块变更后 | 运行时断言和不变量检查 |
| `verify fuzz` | 后期阶段 | 模糊测试，发现边界情况 |
| `verify full` | 最终评测 | 含隐藏测试的完整验证 |

测试矩阵不是人工编写的，而是从学生的本地 `spec/`（架构、模块、操作、目标）和云端 `VerificationPolicy` 自动派生出来的。拆成三块：Base Tests 负责基础公共验证，Design-Driven Tests 从学生自己的架构设计派生，Goal-Specific Tests 从个性化目标派生。

串口标记协议（如 `[SPECLAB] kernel_init`、`[SPECLAB] memory_ready`）让自动验证能精确识别执行阶段。每次运行都产出完整的可复现证据包（`manifest.json`、`events.jsonl`、artifact 引用），验证结果可追溯、可复核。

**教师视角**：

1. **减轻评分负担**：自动验证扛掉了大量重复性检查，教师把精力放在设计审查和个性化指导上。
2. **防止"考试型通过"**：公开、隐藏、fuzz、不变量多层验证合在一起，学生系统不是只在教师预设的用例上能跑，而是在更宽的条件下也能正确运行。
3. **过程性评价有据可查**：证据体系提供了学生开发全过程的客观数据，评价可以基于过程，而不只是最终产物。
4. **科研训练价值**：后续阶段引入的形式化验证方法（模型检查、局部形式化证明），让实验具备了科研训练的属性。

**学生视角**：多层次验证对学生的意义不只是"多跑几个测试"。公开验证给出即时反馈，像一位不用睡觉的助教，每次 push 都告诉你哪里过了哪里没过。隐藏验证保证你的系统不是"刚好跑通了教师给的用例"。fuzz 测试和运行时断言暴露那些你自己根本没想到的边界情况。这种验证密度在传统实验里是不可想象的。传统实验中，学生通常只跑教师给的那一组测试，过了就交，而且往往是在截止时间前最后一刻才跑。本方案的验证是持续性的，早期发现的问题修复成本远低于期末才发现的问题。

### 3.3 AI 辅助体系：受控协作，不是自由代写

本方案对 AI 的使用有整套治理框架。核心定位是"受控协作开发者"，不是"自由代码生成器"。

**Agent 身份与能力模型**：定义了 7 个正式 Agent 身份，每个身份绑定唯一的能力包（CapabilityPack）：

| Agent 身份 | 职责 | 允许的工具类 |
|-----------|------|------------|
| `spec-author.v2` | 编写和审查规格文档 | 读取文档和规格，写入规格目标 |
| `implementer.v2` | 从已批准规格生成代码 | 读取项目上下文，写入 `codegen.targets` |
| `debugger.v2` | 诊断构建、运行和验证失败 | 读取日志、证据和源码，执行诊断命令 |
| `reviewer.v2` | 审查补丁、证据和风险标记 | 读取补丁、规格和证据，不写入代码 |
| `reporter.v2` | 生成报告和证据映射 | 读取证据、规格和报告输入，仅写入报告目标 |
| `toolchain-author.v2` | 维护构建、运行和调试语义 | 写入工具链规格，执行构建生成 |
| `knowledgebase.v1` | 帮助阶段限定的设计决策 | 读取知识库、公开规格和证据，不写入工作区 |

硬约束有四条：写入操作必须绑定本地 spec、SpecPatch ID 或 `codegen.targets`；关键改动必须触发最小验证集；Agent 不得删除测试、关闭检查器或绕过权限；所有 Agent 行为进入审计日志。

**教师视角**：

1. **AI 使用可控可审计**：教师能看到每个学生用了哪些 Agent 功能、在哪个阶段、产出了什么。既防滥用，也为 AI 辅助教学研究提供了数据。
2. **分阶段开放能力**：通过 StageGate，教师可以控制在 boot 阶段只开放 `spec-author` 和 `knowledgebase`，到 memory 阶段再开放 `implementer`。能力随学习进度释放。
3. **学术诚信有据可查**：从 Agent 会话到工具调用到代码变更到验证结果，一整条审计链路贯通，教师能区分学生自主工作和 AI 辅助工作。

#### 3.3.1 可视化：把系统执行流程变成可观测的

Agent 通过 `debug-trace-agent.v1` 和 `qemu-monitor` 技能提供运行时可视化：

- 在关键路径自动插入追踪探针，捕获函数调用链、状态转换和资源流转
- 通过 QMP（QEMU Machine Protocol）获取寄存器状态、内存映射、页表遍历等运行时信息
- 支持符号级 GDB 调试，自动分析调用栈和变量状态
- `vos debug explain-log` 用 Agent 自动分析 QEMU 串口日志、panic 信息和断言失败，把底层错误映射回高级设计概念

**教师视角**：可视化不只是帮学生理解自己的系统。教师可以用 trace 功能在课堂上演示不同调度策略的实际效果、页表遍历过程这些抽象概念，把操作系统的黑盒变成白盒。

#### 3.3.2 知识库：用可审计引用削减 AI 幻觉

`vos-kb` 模块是本方案对抗 AI 幻觉的核心机制：

- 支持课程讲义、实验手册、规格文档、公开证据、参考源码（Git 仓库）、标准文档、网页快照等多种来源
- 基于 `sqlite-vec` 做本地向量索引，用 OpenAI 兼容的 embedding 模型
- 每次 Agent 回答必须附带来源引用（`source_id`、`title`、`object_ref`），学生和教师都可以追溯信息的真实出处
- 学生可以把自己收集的参考源码和资料加入知识库，问答时得到明确来源
- 知识库是项目本地的（`.vos/kb/`），不同学生互不干扰

**教师视角**：教师可以把课程讲义、实验手册作为 `course` 类源加入知识库，确保学生拿到的是经过审核的知识。强制引用让教师能检查学生有没有被 AI 的错误信息带偏。学生的搜索和引用模式本身也是教学数据，能反映他们的学习路径和困难点。

#### 3.3.3 插桩调试：自动暴露内部执行细节

`debugger.v2` 和 `debug-trace-agent.v1` 身份承担了自动插桩调试的工作：

- Agent 读取构建日志、QEMU 输出和 panic 信息，自动推断失败原因
- 在疑似问题路径自动插入诊断代码（打印关键变量、检查不变量），重新构建运行后分析输出
- 插桩调试在独立的 Git worktree 中进行，不污染主工作区
- 确保插桩覆盖了所有可疑路径，调试完成后自动清理

**教师视角**：传统实验里，教师和助教要花大量时间帮学生排查内存越界、页表配置错误这些常见问题。自动插桩调试把这部分重复劳动接过去了，教师能把时间用在设计层面的指导上。

## 四、vos 工具链

### 4.1 全流程可追溯

vos 工具链把从开发到评测的每一步都留下了可追溯的记录。

**Git Commit 严格绑定**：每次 Agent 代码生成操作必须从 clean tree 开始，生成后自动创建 VOS 管理的 Git commit。commit 带 `.vos/commit-ledger.jsonl` 条目，记录 `commit_sha`、`parent_sha`、`actor`（agent 还是 human）、`agent_identity_id`、`run_id`、`spec_refs`、`evidence_refs` 等信息。人工 commit 也要写明 `collaboration_intent` 和是否基于 Agent 输出。代码演化形成完整的 DAG，每一行都可以追溯到它产生的背景。

**运行证据保存**：每次 `vos` 命令执行都有唯一 `run_id`，在 `.vos/runs/<run-id>/` 下保存 `manifest.json`（命令、状态、时间、产物）、`events.jsonl`（事件流水）和 `artifacts/`（构建日志、QEMU 日志、验证摘要等）。验证复现以 `commit_sha` 为唯一输入锚点。

**教师视角**：

1. **过程性评价有客观基础**：教师能追溯学生的完整开发历程，什么时候完成了哪个阶段、遇到了什么错误、怎么解决的，而不只是看最后一次提交。
2. **学术诚信可审查**：commit ledger 和 Agent 审计日志合在一起，能准确判断代码是学生自己写的还是 AI 生成的，AI 辅助到了什么程度。
3. **指导有数据支撑**：从学生的开发轨迹里识别薄弱环节（比如反复在某个模块失败），提供针对性指导。

### 4.2 工具无关的语义构建规范：ToolchainSpec

`ToolchainSpec` 把构建系统的"做什么"和"怎么做"拆开了。它是工具无关的语义规范：

- 学生只维护一份语义 spec（源文件、编译标志、链接约束、依赖关系）
- Agent 可以为不同场景起草不同工具链实现（Makefile、CMake、xtask、Bazel 等）
- `vos-cli` 执行确定性 gate 校验（路径、spec hash、manifest、ledger、dry-run），确保物化出来的构建系统不偏离语义规范
- 构建环境约束（工具版本、target triple、ABI）结构化记录在 `.vos/toolchain.json` v2 中，`vos build` 执行前自动探测并校验

**教师视角**：传统实验里"在我电脑上能跑"是最常见的教学障碍。ToolchainSpec 的结构化环境约束和自动探测机制把环境一致性问题从学生手上接了过来。学生专注于构建语义（"我的内核需要哪些源文件、什么编译标志"），不必为 Makefile 语法分心。同一份 ToolchainSpec 可以适配不同工具链后端，实验方案不绑定特定构建工具版本。

### 4.3 统一命令入口与策略执行

`vos` 把所有实验操作收进了固定子命令（`vos build`、`vos run qemu`、`vos verify public` 等），不允许任意 shell 转发。策略白名单控制 Agent 能执行什么命令、能访问哪些路径。Portal-bound repo 要做在线身份校验和 policy snapshot gate。所有命令输出稳定 JSON 格式，方便平台消费。

## 五、指导手册与教学设计创新

### 5.1 指导手册的定位创新：设计导航而非操作手册

传统实验指导书通常是"跟着做就能得到正确答案"的操作手册——第一步装什么软件，第二步改哪个文件，第三步跑什么命令。学生按图索骥完成操作，但往往不知道"为什么要这么做"。VeriSpecOSLab 的实验指导书选择了完全不同的定位：它是一份**设计导航**。

指导书的核心约束有三条。第一，**描述设计问题，不预设实现方案**。指导书会告诉你"这个阶段必须解决资源命名和生命周期管理的问题"，但不会告诉你"用 file descriptor table"。第二，**定义质量门禁，不指定通过方式**。指导书会告诉你"分配器不能返回已被分配的页"，但不会告诉你"用 freelist 还是 buddy"。第三，**要求设计理据，不接受"就是这样"**。每个设计决策都要记录在 ArchitectureSlice 或 ADR 中：你借鉴了什么？修改了什么？拒绝了什么？为什么？

指导书采用"一站四册"结构：Book（教材型章节，描述每个阶段的设计空间、背景知识和典型路线）、Labs（实验卡片，列出设计问题、规格要求、质量门禁和提交物）、Specs 手册（规格写作手册，教学生怎么写 ArchitectureSeed、ModuleSpec、OperationContract）、Appendices（工具参考：vos 命令、QEMU、GDB、RISC-V、AI 策略等）。四个部分各司其职——Book 解释"为什么"，Lab 说明"做什么"，Specs 手册教"怎么写"，Appendices 提供"怎么查"。

**教师视角**：这种结构化手册让教师从"反复回答学生的基础操作问题"中解放出来。学生遇到工具问题查 Appendices，遇到设计困惑查 Book，遇到规格格式问题查 Specs 手册。教师和助教的答疑时间可以集中在真正的设计讨论上。手册的三条核心约束还给了教师一个明确的审核框架：检查学生是否回答了设计问题（而非照搬了某个参考实现）、是否通过了质量门禁（而非绕过了检查）、是否记录了设计理据（而非写了"就是这样"）。

**学生视角**：传统指导书的最大问题是"你不知道自己错过了什么"——它只告诉你一条路，你走到终点以为这就是唯一的路。设计导航型指导书把设计空间摊在你面前：这里有三条路，每条路的历史、代价和适配场景都在这里。你选哪条？为什么？这种训练让学生在走出校门后面对真实工程问题时，不会陷入"没有参考答案就无法决策"的困境。

### 5.2 历史驱动的教学设计：让每一个设计决策有来历

VeriSpecOSLab 的指导手册将操作系统历史深度嵌入教学设计，不是作为"课外阅读"，而是作为理解每个阶段设计问题的必要背景。

**从历史讲原理，而不是从定义讲原理。** 以内存管理为例。传统教材的开头通常是"虚拟内存是指将辅存当作内存来使用的技术，它使得程序能够使用比物理内存更大的地址空间"。本指导书第 3 章的开头是：1961 年，曼彻斯特大学的 Atlas 计算机首次实现了虚拟内存。在此之前，程序员必须手动管理物理内存位置。"我的数据放在 0x2000 到 0x3000，你的程序不能碰那里"——这种约定在批处理时代勉强可行，在多道程序时代完全不可控。Atlas 的洞见是让硬件做地址翻译，由此产生了两个深远影响：隔离（程序 A 和 B 可以用相同虚拟地址，但映射到不同物理页）和虚拟化（物理内存不够时把不用的页换到磁盘上）。学生在理解了"为什么需要虚拟内存"之后，再学 Sv39 页表的三级结构、TLB 刷新、page fault 处理，就不觉得这是从天而降的复杂机制。

**用历史争论揭示设计权衡。** 以进程创建为例。指导书第 5 章花了大量篇幅讲 Unix 的 fork 是怎么来的：1971 年，PDP-7 只有 8 KB 内存，Ken Thompson 和 Dennis Ritchie 做了一个简化——"复制当前进程，然后让子进程自己去替换程序"。他们以为这是临时方案，但后来发现了出乎意料的好处：fork 和 exec 之间的窗口让 Shell 能偷偷修改子进程的文件描述符和环境变量，而子程序对此毫无感知。这就是 Unix 组合性的基石。但指导书同时也讲 fork 的代价：在 2020 年代复制地址空间是巨大开销，Copy-on-Write 虽然缓解了问题，却也带来了引用计数和 page fault handler 的复杂度。Windows NT 的 CreateProcess 选择了 spawn 一步模型，避免了 fork 的大多数开销，但失去了窗口期的灵活性。两种选择都合理——指导书不替学生做选择，而是把两种选择的来历和代价都摆在面前。

**引入安全事件作为教学锚点。** 第 5 章用了大量篇幅讲述 Meltdown 和 Spectre——不是作为"安全课的附录"，而是作为"你为什么需要页表隔离"和"为什么边界检查还不够"的教学锚点。2018 年 1 月，Meltdown 漏洞曝光：Intel CPU 在推测执行时不检查页表的 U/S 权限位，攻击者可以在用户态逐字节读出全部内核内存。Linux 社区的应对是 KPTI——每个进程维护两份页表，用户态运行时页表里几乎没有内核映射。这个事件让"页表隔离"从"可选的微内核特性"变成了"所有主流 OS 的标配"。Spectre 则更进一步——它攻击的是分支预测器本身，在同一个地址空间内就能完成攻击，页表隔离对它完全无效。这两个故事合在一起给了学生一个深刻的教训：地址空间隔离不能只依赖权限位（Meltdown 的教训），也不能只依赖页表（Spectre 的教训）。

**教师视角**：历史驱动的教学设计改变了课堂叙事。教师不是在讲"虚拟内存是这样定义的"，而是在讲"1961 年 Atlas 上的程序员面对一个问题，他们想出了这个解决方案，六十年后你的 Sv39 页表是它的直系后代"。这种叙事让学生意识到操作系统的每个机制都是人类决策的产物，不是从天而降的数学定理。安全事件（Meltdown/Spectre）的引入则提供了另一种教学价值：它展示了"真实世界的工程约束如何改写教科书"——2018 年之前所有 OS 教科书都默认内核和用户共用页表是理所当然的，2018 年之后这个默认被推翻了。

**学生视角**：知道 fork 是 PDP-7 硬件限制下的临时方案，比背诵"fork 创建一个子进程，返回两次"更有助于理解 fork 的设计哲学。知道 Meltdown 如何用 cache 侧信道泄漏内核数据，比单纯记住"KPTI 会降低 syscall 性能 5-30%"更能理解页表隔离的设计动机。历史叙事把抽象概念变成了有起因、有冲突、有后果的故事——而人类大脑天生善于记忆故事。

### 5.3 分层挑战设计：零基础与有经验学生各得其所

指导书在每个章节中标注了"挑战路线"提示，面向有 OS 开发经验或追求更深设计理解的学生。这些挑战问题不是"额外加分项"，而是引导有基础的学生跳过他们已经掌握的内容，把精力投入更有深度的设计问题。

例如，阶段 1 的挑战问题包括：如果选微内核，IPC 性能如何保证（零拷贝 IPC、syscall 批处理、快速路径优化）？如果选 Exokernel，libOS 边界怎么划（内核越薄，libOS 之间的互操作性越差，这个张力怎么解决）？如果选 Rust 写内核，unsafe 的边界在哪里（每个 unsafe 块应最小化，并在文档注释中解释为什么编译器不能证明这段代码的安全性）？阶段 8 的方向组合前瞻（你现在的设计决策如何影响未来的扩展空间）？

这种设计的核心思路是：同一份指导书，不同学生读到的是不同的深度。零基础学生按主线走，聚焦"理解原理、完成基本实现"。有经验学生在主线之外还可以深入挑战问题，把同一阶段做出不同的高度。

**教师视角**：分层挑战解决了班级学生基础差异大的教学难题。教师不需要为"吃不饱"和"跟不上"的学生分别准备两套材料。零基础学生完成主线要求即可通过阶段门禁；有经验学生的挑战产出（如微内核 IPC 优化方案、Exokernel libOS 边界设计）可以体现在 ADR 和个性化目标中，作为评分中的加分依据。

**学生视角**：对零基础学生，挑战路线的存在本身就是一种信号——"你现在做的不是这个阶段的全部可能性，前面还有更有意思的东西等着你"。对有经验学生，挑战路线让他们免于在早已掌握的基础内容上浪费时间，可以把同样的课程学分投入到更有深度的工作中。

### 5.4 跨 ISA 对比与语言选择引导

指导书不仅支持 RISC-V，还在各章节中以对比表格的形式标注了 RISC-V、ARM 和 x86-64 三大 ISA 的关键差异。例如启动链对比（RISC-V 的 OpenSBI 直启 vs x86-64 的 GRUB/Multiboot2 vs UEFI 直启）、特权级对比（RISC-V M/S/U 三级 vs ARM EL3-EL0 四级 vs x86 Ring 0-3）、页表对比（Sv39 三级 vs VMSAv8-64 四级 vs 4-level PML4）、中断控制器对比（PLIC vs GICv3/v4 vs APIC/x2APIC）。

指导书还对系统编程语言选择提供了详细分析：C（语法简单但 UB 陷阱隐蔽，参考资料最丰富）、Rust（编译期消灭内存 bug 类别，但所有权学习曲线陡峭）、Zig（comptime 编译期代码执行是独特优势，交叉编译体验一流）。每种语言从内存安全、代表项目、构建系统、交叉编译、学习曲线、适合场景六个维度做了对比。

**教师视角**：跨 ISA 和跨语言对比让实验方案不绑定特定技术栈。不同学校可以根据自己的硬件条件和教学传统选择 ISA 和语言，而指导书的核心设计方法保持不变。同一班级内也可以允许不同学生选择不同 ISA 或语言，只要他们在 ArchitectureSeed 中声明并论证选择。

**学生视角**：学生在选择 ISA 和语言时不是在黑暗中做决定。对比表让他们看到每个选项的代价和优势。一个学生在看过对比后选择"RISC-V + Rust"，不是因为教师说了这是最好的选择，而是因为 TA 理解了 Rust 的所有权模型能帮自己少写多少内存 bug、RISC-V 的简洁 ISA 规范能让自己更快看懂特权手册。这种有理由的选择本身就是系统设计训练的一部分。

### 5.5 递进式阶段门禁

本方案定义了清晰的递进阶段：

```
architecture-seed → boot-minimum → memory-management → trap-privilege
→ execution → syscall-ipc → resource-and-namespace → final-synthesis
```

每个阶段的门禁（StageGate）包含必需产物、自动检查（`spec lint`、`arch lint`、`build`、`verify public` 等）、教师或助教对架构设计的人工审核，以及解锁条件。没通过前置阶段，后面阶段不开放。

**教师视角**：

1. **防止跳跃式学习**：boot 没过就不能进 memory，学习路径是受控的。
2. **早期干预窗口**：如果学生在 architecture-seed 阶段就暴露了设计思维问题，教师可以在这里纠正，不让错误传播到后面所有阶段。
3. **灵活可调**：平台允许课程调整具体阶段名和顺序，阶段门禁的语义是统一的。

### 5.6 个性化目标合约

学生可以定义自己的个性化目标，通过 `GoalValidationContract` 形式化表达。目标可以是二进制兼容（与某个已有 OS 的 ABI 兼容）、性能优化（系统调用延迟、IPC 吞吐量、文件系统 IOPS）、硬件移植（从 QEMU 迁到真实 RISC-V 开发板）、安全隔离（Capability 机制或命名空间隔离）、可验证性增强（引入形式化方法），以及镜像体积和启动速度优化等。

每个目标合约必须包含基线（baseline）、目标（target）、正确性保护（correctness_guard，防止为追求数字破坏正确性）、评判标准（benchmark_or_oracle）和必需证据（evidence_required）。

**教师视角**：不同学生的目标反映了他们的兴趣方向，教师可以据此提供更精准的指导。基线、目标和正确性保护合在一起，为不同方向的评分提供了公平的比较基准。比的不是谁的 OS 跑得快，而是谁在自己的目标方向上进步最大。个性化目标机制还让实验具备了初步的科研训练属性，学生需要定义研究问题、设定评估标准、收集证据。

### 5.7 教学分析看板（计划内）

平台设计了一套教学分析框架，涵盖阶段通过率分布（识别教学难点）、常见失败类型排行（boot 失败、内存管理错误、系统调用错误等）、设计审核中的高频问题、AI 使用强度分布、目标选择偏好，以及高风险项目自动识别（长期卡在同一阶段、AI 大量生成但解释不足、设计与实现长期不一致等）。本地证据采集和审计日志已完整可用；看板的聚合展示和风险自动识别仍有待建设。

**教师视角**：教师可以根据数据识别班级整体和个别学生的薄弱环节，调整教学策略。多轮课程的数据对比帮助持续优化实验设计，比如某个阶段通过率持续偏低，就可能需要调整前置教学内容或阶段难度。AI 使用数据本身也是研究 AI 辅助教学效果的客观依据。

## 六、平台架构

### 6.1 SpecLab / CaseLab 通用平台（部分计划内）

VeriSpecOSLab 的规格驱动方法和 Agent 架构不限于操作系统。凡是具备"系统状态复杂、模块边界清晰、正确性可由规格描述、可以通过测试或 benchmark 评估、需要 AI 辅助但不能脱离约束"这五个特征的实验，都能放进同一个平台框架。本方案定位为通用的 **Spec-Driven Lab Platform**（亦可称 **CaseLab**），操作系统是首个落地领域，已完整实现。数据库、编译器、网络、运行时、体系结构等方向的插件适配器仍在设计中。

**可扩展的实验领域**：

| 领域 | 实验方向举例 | 核心规格关注点 | 主要验证手段 |
|------|------------|-------------|------------|
| **SpecOSLab** | 宏内核、微内核、Unikernel | 并发、隔离、硬件状态、中断、系统调用 | QEMU 启动、trap trace、不变量检查、syscall 测试 |
| **SpecDBLab** | KV Store、关系数据库、LSM-tree、B+Tree 引擎、事务数据库、分布式数据库 | 持久化、事务隔离、恢复、索引不变量、页面锁存 | oracle 对比、崩溃 fuzz、事务隔离测试、benchmark |
| **SpecLangLab** | 词法/语法分析、类型检查、IR 优化、代码生成、JIT 编译 | 语义保持、IR 不变量、类型规则、pass 前置条件 | 差分测试、IR verifier、解释器等价、ABI 测试 |
| **SpecNetLab** | TCP-like 协议、可靠传输、路由、RPC、QUIC-like | 协议状态机、包格式、超时/重传、安全属性 | 模型检查、包 trace 对比、fuzz、互操作测试 |
| **SpecRuntimeLab** | 字节码 VM、GC、JIT、协程运行时 | 对象模型、字节码安全、GC 不变量、栈帧布局 | 字节码 verifier、解释器/JIT 差分测试、GC 压力测试 |
| **SpecHWLab** | ISA 模拟器、流水线 CPU、cache、MMU、总线 | ISA 语义、流水线不变量、内存一致性、cache 一致性 | trace 对比、参考仿真器差分、形式化模型检查 |

**领域适配机制**（计划内）：平台核心（Agent Gateway、DevBox、测试矩阵、报告生成、评分框架、审计日志、个性化目标合约）对所有领域通用——其中 Agent Gateway、DevBox、测试矩阵、审计日志和个性化目标合约已在 OS 领域落地。每个领域需要补充三层插件：领域规格模板（定义该领域的 ModuleSpec 字段语义）、领域验证 oracle（QEMU、数据库 replay、差分编译器、协议模型检查器等）和领域 Agent 身份（SpecDBAgent、SpecLangAgent 等）。插件架构和适配器接口尚未实现。

以 SpecDBLab 为例（计划内），B+Tree 索引模块的规格需要额外定义页面锁存协议、分裂/合并的并发安全、WAL 与脏页刷盘的持久化约束，以及崩溃恢复场景的测试义务。SpecLangLab 的类型检查模块（计划内）则要定义 AST 到 TypedAST 的映射、符号表解析规则、类型不匹配的诊断输出格式，并用解释器等价性作为跨 pass 的验证 oracle。这些领域特化字段通过 `ModuleSpec` 的泛化模板统一承载，平台核心逻辑不感知领域差异。

**学生视角**：对于学生，SpecLab 意味着在操作系统课程中学到的"规格驱动开发"方法论可以迁移到后续任何一门系统类课程。一个在 OS 课上学会了写 `OperationContract` 的学生，到了数据库课上不需要重新学习怎么描述 B+Tree 的 kalloc 等价操作，到了编译器课上也不需要重新学习怎么为常量传播写 `preconditions` 和 `postconditions`。工具是同一套 `vos`，流程是同一条 `spec → agent → build → verify → evidence` 闭环，学习曲线从第二门课开始大幅降低。更长远地看，学生在每门课积累的规格文档、ADR 和证据包本身就是一份越来越厚的系统工程 portfolio。

**教师视角**（平台层计划内）：对开设多门系统类课程的院系，统一平台减少了不同课程各自维护实验环境的重复投入。更长远的好处是教学数据的跨课程共享。数据库课的老师能看到学生在 OS 课上的架构设计质量，判断他们对并发和持久化的前置理解到了什么程度；编译器课的老师能看到学生在数据库课写的 IR 优化相关代码，决定从哪里开始讲 SSA。本地工具链和单课程实验流程已可投入使用；跨课程数据贯通和平台层多租户管理尚未建成。

### 6.2 三层可见性投影（本地端已实现，云端管理层计划内）

同一份课程规则按三种可见度投影给不同角色：`student-public`（公开要求、评分项、阶段门禁、反馈）、`agent-public`（在 student-public 基础上增加 agent-only 风险标签）、`staff-full`（全部课程规则、隐藏验证和评分细节）。本地工具链和 Agent 的可见性过滤已落地，学生端和 Agent 端接触不到隐藏测试细节，Agent 能拿到足够上下文但读不到 staff-only 敏感信息。云端的规则管理和投影分发服务待建设。

### 6.3 Local-First 架构（核心已实现，云端平台层计划内）

学生本地运行 `vos-agent`（OpenAI-compatible API 服务）和 `vos-cli`，这两部分已完整可用。云端平台作为 control plane 的角色（签发身份、项目绑定和 policy snapshot，调度 sandbox runner，归档证据和审计记录）尚在设计中。Agent tool execution 已实现为永久 local-only，云端不承载 workspace Agent 执行。隐藏测试、mutation 规则和 staff-only 评分细节始终不进学生仓库。

**教师视角**：学生代码、设计文档和学习数据主要留在本地，数据隐私天然有保障。学生各自使用自己的 LLM 提供者，教师不需要统一采购 API 额度。本地 Agent 架构让学生在断网时仍能做大部分工作（仅 Portal-bound 受控命令需在线校验）。

### 6.4 开放 IDE 接入

本方案支持学生通过 OpenAI-compatible API 接入多种 IDE 和工具：VS Code、JetBrains、Cursor、Continue、Cline、Aider、Open WebUI、CodeGPT 等。在本地配置 `base_url = http://localhost:8080/v1` 即可接入 `vos-agent`，学生可以用自己习惯的开发环境。

### 6.5 跨课程连续化教学（计划内）

SpecLab 的跨领域设计背后有一个更值得做的教学创新：把原本各自独立的多门系统类课程串成一条连续训练链。当前 OS 实验的完整工具链和本地 Agent 已经跑通；多课程的连续化链路和跨课程数据贯通还在设计中。

设想一个学生完整的本科系统课程路径：

```
OS 课程（SpecOSLab）
  → 学生实现 kernel / syscall / fs / IPC，定义自己的 ABI

编译器课程（SpecLangLab）
  → 学生实现编译器，目标后端可以是自己 OS 的 ABI
    编译器产出的可执行文件能跑在自己写的内核上

数据库课程（SpecDBLab）
  → 学生实现 KV Store 或关系数据库，运行在自己 OS 或 Linux 上
    数据库的持久化层直接调用自己写的文件系统接口

网络课程（SpecNetLab）
  → 学生实现 TCP-like 协议栈或 RPC 框架，运行在自己 OS 上
    协议栈复用自己写的网卡驱动框架

体系结构课程（SpecHWLab）
  → 学生实现 RISC-V 模拟器或 FPGA softcore，再把 OS 移植上去
    OS 第一次跑在"自己的硬件"上

分布式系统课程
  → 学生基于自己之前的 OS、网络栈和数据库，实现分布式 KV 或 Raft
```

每门课的成果不再是学期结束就被丢弃的一次性作业。OS 课上定义的 syscall ABI，在编译器课上成为目标后端；数据库课上的文件系统接口，直接对接 OS 课上的 VFS 实现。学生在四年里积攒的不只是学分，而是一套从硬件到分布式应用、自己亲手写出来的完整系统栈。

**学生视角**：这条路径最大的吸引力在于"连续性"。传统的系统类课程各自独立，每门课都要重新搭环境、重新学工具、重新适应一套规则。OS 课上用 xv6 框架，数据库课上用 MySQL 源码，编译器课上用 LLVM，工具链和实验范式完全不同。SpecLab 把实验环境收进了同一套 `vos` 命令、同一套 spec 规范、同一套 Agent 协作方式。学生在第二门课上花在"熟悉平台"上的时间几乎为零，精力可以全部投入领域知识本身。毕业时，这套完整的系统栈 portfolio 不仅是求职材料，更是对自己四年所学的一次系统性验证：从 kernel 到 compiler 到 database 到 network，每一层都亲手写过、验证过、理解过。

**教师视角**（计划内）：跨课程连续化要解决的是系统类课程群的两个老问题。一是"前置依赖不透明"。数据库老师不知道学生在 OS 课上学过什么程度的并发控制，编译器老师不知道学生对 ABI 的理解到了哪个层次。跨课程数据贯通可以让前置知识变得可查询。二是"课程孤岛"。每门课独立实验、独立评分、独立工具链，学生感受不到这些课之间的联系。连续化训练链让学生看到操作系统、编译器、数据库、网络这些"系统"是怎么拼在一起工作的。这种全局视角，单独哪门课都给不了。当前各课程的独立实验流程已可运行，跨课程数据贯通和训练链编排层尚未建成。

| 维度 | 传统 OS 实验 | VeriSpecOSLab |
|------|------------|---------------|
| **设计自由度** | 单一内核形态，框架限定 | 学生自主设计架构，支持多种内核形态 |
| **开发方式** | 手动编写代码，补全框架 | 规格驱动，Agent 辅助生成和验证 |
| **学习重心** | 代码实现细节 | 系统设计思维、架构权衡、正确性定义 |
| **AI 使用** | 无约束或完全禁止 | 受控协作，绑定规格，完整审计 |
| **验证方式** | 固定测试用例 | 多层级自动验证（公开+隐藏+fuzz+不变量+形式化） |
| **可追溯性** | 仅最终代码提交 | 全流程 Git commit ledger + 运行证据 |
| **个性化** | 所有学生同一目标 | 个性化目标合约，学生自选优化方向 |
| **教师角色** | 出题者+评分者 | 设计导师+过程评估者+数据分析者 |
| **可扩展性** | 仅限操作系统 | 通用 SpecLab/CaseLab 平台，支持数据库、编译器、网络、运行时、体系结构等 |
| **环境一致性** | 依赖学生自行配置 | ToolchainSpec + DevBox 保证环境可复现 |
| **知识保障** | AI 幻觉无防护 | 知识库+强制引用+可审计来源 |

将上述创新放到更长的时间尺度上看，SpecLab/CaseLab 的跨领域和跨课程连续化设计让它的价值不止于一门操作系统实验课。当前，OS 实验的完整本地工具链（vos CLI + vos-agent + vos-kb + vos-spec）已全部可用，学生可以完整体验从架构设计、规格编写、AI 辅助生成、自动验证到证据追溯的全流程。平台层（云端 Portal、多领域插件、跨课程数据贯通、教学分析看板）和跨课程连续化训练链仍在建设中。完整落地后，它提供的是一个从大一到大四、从 OS 到分布式系统的系统能力训练框架。学生在四年里用同一套方法和工具，在不同领域反复练习"规格 → 设计 → 实现 → 验证"的闭环，最终积累的不是几门课的成绩，而是一套亲手构建的、从硬件到应用层的完整系统栈。对教师而言，这个框架把散落的系统类课程串成了一个可观测、可追溯、可迭代优化的教学连续体。这不是换了一套实验工具，是换了一种教和学的关系。

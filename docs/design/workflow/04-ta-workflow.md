## 1. 角色职责

助教负责：

- 维护批量审核队列
- 对公开失败做一线归因
- 触发基础设施异常补跑
- 协助学生理解公开反馈与阶段要求
- 收集高风险项目并升级给教师
- 处理申诉的事实核查部分

助教不负责：

- 修改课程规则
- 发布最终成绩
- 单独决定高风险项目的最终结论

## 2. 助教可见与不可见

助教可见：

- 本项目公开反馈摘要
- staff 诊断摘要
- hidden suite 的受控结果与失败分类
- 审计风险标签
- 历史补跑与审核记录

助教不可做：

- 向学生暴露 hidden tests 细节
- 直接绕过教师定义的门禁
- 口头承诺与系统记录不一致的 override

## 3. 日常主线

```text
查看审核队列
  -> 区分自动通过 / 待人工审核 / 失败待归因
  -> 对公开失败做初步判断
  -> 必要时补跑或要求学生补充材料
  -> 记录处理结果
  -> 对高风险案例升级给教师
```

## 4. 失败归因

助教对失败至少区分三类：

- 学生设计或实现错误
- 基础设施错误
- 规则或课程配置问题

处理原则：

1. 若是学生错误，只返回公开摘要和修复方向，不泄露 hidden 细节。
2. 若是基础设施错误，触发补跑并保留原始失败 run。
3. 若怀疑规则问题，收集证据后升级给教师。

## 5. 补跑与异常处理

助教可以触发：

- 单项目补跑
- 指定阶段补跑
- 指定验证套件重跑

补跑前必须确认：

- 原始失败属于基础设施异常或环境污染
- 当前项目未被冻结
- 补跑范围不会破坏证据链

典型基础设施异常包括：

- runner 宕机
- QEMU 启动环境故障
- cache 污染导致的非确定性失败
- 日志采集或 artifact 上传失败

## 6. 答疑与申诉

助教答疑时只能依赖：

- 当前阶段公开要求
- 公开失败摘要
- 学生自己的 spec / code / report

助教不能给出：

- hidden suite 的具体触发条件
- 其他学生项目实现
- 未记录在系统中的口头豁免

申诉处理中，助教负责：

- 收集 run、日志、评分摘要和补充说明
- 初步判断是否为规则、证据或 infra 争议
- 把需要裁决的案例升级给教师

## 7. 升级条件

出现以下情况必须升级：

- 多次补跑后仍表现为疑似 infra failure
- AI 审计显示越权、删除测试或大规模未解释生成
- hidden suite 与公开结果长期矛盾
- 学生要求修改正式成绩或挑战课程规则
